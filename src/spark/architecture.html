Spark Architecture

Cluster Architecture

Spark uses a master-worker architecture. The Driver coordinates work across the cluster, while Executors run tasks on Worker Nodes.

+------------------+     +------------------+
|    DRIVER        |     | CLUSTER MANAGER  |
| (Your Python     |<----| (YARN/K8s/      |
|  Code)           |     |  Standalone)     |
+--------+---------+     +--------+---------+
         |                        |
         | allocates              | allocates
         v                        v
+--------+---------+     +------------------+
|   EXECUTORS      |<----|  WORKER NODES    |
| (JVM processes)  |     | (Multiple per    |
| - Runs tasks     |     |  machine)        |
| - Stores cache   |     +------------------+
+--------+---------+
         |
         v
+------------------+
|   STORAGE        |
| (HDFS/S3/Local) |
+------------------+

Key Components

  • Driver: Runs your main() function, creates SparkContext, coordinates tasks
  • Cluster Manager: Allocates resources (YARN, Kubernetes, Standalone)
  • Worker Node: Machine in the cluster that runs executor processes
  • Executor: JVM process that runs tasks and stores data in memory
  • Task: Smallest unit of work scheduled on an executor

The DAG (Directed Acyclic Graph)

When you write PySpark code, Spark builds a DAG of transformations. Only when you call an Action does Spark execute the DAG.

User Code: df.filter().select().groupBy().agg()
                    |
                    v
         +-------------------+
         |   LOGICAL PLAN    |
         | (What to compute) |
         +-------------------+
                    |
                    v
         +-------------------+
         |  PHYSICAL PLAN    |
         | (How to compute)  |
         +-------------------+
                    |
                    v
         +-------------------+
         |  RDDs (Partitions)|
         | (Data distribution)|
         +-------------------+
                    |
                    v
         +-------------------+
         |    EXECUTION      |
         | (Tasks on cluster) |
         +-------------------+

Lazy Evaluation

Transformations are lazy - they build a plan but don't execute until an action is called. This allows Spark to optimize the entire plan.

# These lines build a plan but do NOT execute:
df2 = df.filter(col('salary') > 50000)
df3 = df2.select('name', 'salary')
df4 = df3.withColumn('bonus', col('salary') * 0.1)

# NOW it executes:
df4.show()  # Action triggers the full DAG

Executor Memory Architecture

Each executor has memory divided into regions for storage and execution:

Region Purpose
Heap Memory Stores DataFrames, cached data, intermediate results
Off-Heap Tungsten memory (serialized data, no GC)
User Memory User data structures, internal metadata
System Memory JVM overhead, OS reserved

Partitions

Data is partitioned across the cluster. Each partition is processed by a single task in parallel.

# Check partitions
df.rdd.getNumPartitions()

# Repartition (full shuffle)
df.repartition(20)

# Coalesce (reduces, no shuffle)
df.coalesce(4)

# Partition by column
df.repartition(10, 'dept')

Shuffle

Shuffle redistributes data across partitions. It's expensive because it involves disk I/O and network transfer.

Causes of Shuffle: groupBy(), join(), distinct(), orderBy(), repartition()
