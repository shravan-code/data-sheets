Table Formats & Lakehouse Layer

7 topics

Apache Hudi

Hudi (Hadoop Upserts Deletes and Incrementals) is an open table format designed for record-level upserts and incremental processing on data lakes. Originally built at Uber for their ingestion use case.

📌

Core Concepts

1

Table Types

💡

Example

Upsertfrom pyspark.sql import SparkSession from pyspark.sql.functions import lit # Upsert new/changed orders into Hudi table new_records.write.format('hudi')\ .option('hoodie.table.name', 'orders')\ .option('hoodie.datasource.write.recordkey.field', 'order_id')\ .option('hoodie.datasource.write.precombine.field', 'updated_at')\ .option('hoodie.datasource.write.operation', 'upsert')\ .mode('append')\ .save('s3://lake/hudi/orders')
Incremental Readspark.read.format('hudi')\ .option('hoodie.datasource.query.type', 'incremental')\ .option('hoodie.datasource.read.begin.instanttime', '20240101000000')\ .load('s3://lake/hudi/orders')
⚠️

Drawbacks

  • •More complex than plain Parquet — requires Hudi-aware reader
  • •MOR tables require compaction scheduling
  • •Smaller community than Iceberg for non-Spark engines
  • •COW writes are expensive on large partitions

Apache Iceberg

Apache Iceberg is an open table format for huge analytic datasets. Designed for correctness and performance at Netflix scale. Key innovation: hidden partitioning and pointer-based metadata for atomic operations.

🏛

Key Features

Featuredescriptionexample
Hidden PartitioningPartition transforms (year, month, bucket, truncate) are applied automatically. Query writers don't need to know partition layout.CREATE TABLE orders PARTITIONED BY (days(order_date), bucket(16, customer_id)); -- No need to filter on year/month separately
Time TravelSELECT * FROM catalog.db.orders FOR SYSTEM_VERSION AS OF 5; SELECT * FROM catalog.db.orders FOR SYSTEM_TIME AS OF '2024-01-01 00:00:00';
Schema Evolution
Row-Level Deletes (GDPR)DELETE FROM catalog.db.customers WHERE customer_id = 12345; -- Works on Iceberg
⚠️

Drawbacks

  • •Catalog management adds operational overhead
  • •Metadata file explosion on large tables with many small commits
  • •Not all engines support all Iceberg features equally (v1 vs v2)
  • •Requires compaction and metadata cleanup (expire_snapshots)

Delta Lake

Delta Lake is an open-source storage layer on top of Parquet files, developed by Databricks. It uses a transaction log (_delta_log) to enable ACID transactions, time travel, and DML on data lakes.

📌

Transaction Log

Locations3://lake/delta/orders/_delta_log/
StructureJSON/Parquet checkpoint files recording every operation
Example Log Entry{"add":{"path":"part-00001.parquet","size":1048576,"modificationTime":1705312200…
📌

Key Operations

1

MERGE (Upsert)

Sql

MERGE INTO delta.`s3://lake/silver/orders` AS target USING new_orders AS source ON target.order_id = source.order_id WHEN MATCHED THEN UPDATE SET target.status = source.status WHEN NOT MATCHED THEN INSERT *;

2

Time Travel

Sql

SELECT * FROM delta.`s3://lake/silver/orders` VERSION AS OF 10; SELECT * FROM delta.`s3://lake/silver/orders` TIMESTAMP AS OF '2024-01-01';

3

Schema Evolution

Spark

df.write.option('mergeSchema', 'true').format('delta').mode('append').save('s3://lake/silver/orders')

4

OPTIMIZE + ZORDER

Sql

OPTIMIZE delta.`s3://lake/silver/orders` ZORDER BY (customer_id, order_date);

5

VACUUM (cleanup old files)

Sql

VACUUM delta.`s3://lake/silver/orders` RETAIN 168 HOURS;

⚠️

Drawbacks

  • •Transaction log grows large on high-frequency writes — needs checkpointing
  • •Primarily optimized for Databricks/Spark — other engines less mature
  • •VACUUM deletes time travel history — must balance retention vs storage cost
  • •Concurrent writes require careful coordination

Time Travel

Time travel lets you query previous versions of a table. Critical for auditing, debugging pipelines, and recovering from accidental deletes.

🎯

Use Cases

  • →Audit: what did the orders table look like before the bad data load?
  • →Recovery: recreate Gold table from Silver snapshot before corruption
  • →ML reproducibility: retrain model on exact dataset version used 6 months ago
  • →Debugging: compare before/after a transformation
📌

Examples

DeltaSELECT * FROM orders VERSION AS OF 42; SELECT * FROM orders TIMESTAMP AS OF '2024-01-01 00:00:00'; SELECT * FROM orders@v42; -- shorthand
IcebergSELECT * FROM db.orders FOR SYSTEM_VERSION AS OF 8473627384; SELECT * FROM db.orders FOR SYSTEM_TIME AS OF TIMESTAMP '2024-01-01 00:00:00';
Hudispark.read.format('hudi').option('as.of.instant', '20240101000000').load('s3://lake/hudi/orders')
⚠️

Drawbacks

  • •Time travel history is limited by VACUUM / snapshot expiry settings
  • •Keeping long history increases storage costs

ACID on Data Lakes

Traditional data lakes (raw Parquet/CSV on S3) have no ACID guarantees. Table formats (Delta/Iceberg/Hudi) add an ACID layer through optimistic concurrency control.

📌

Acid On Lake

AtomicityCommit writes either fully succeed (new snapshot created) or fully fail (no partial files visible)
ConsistencySchema enforcement prevents invalid data from being written
IsolationSnapshot isolation — readers see consistent snapshot even while writes are in progress
DurabilityCommitted data written to durable object storage (S3/GCS); transaction log persists commit history
