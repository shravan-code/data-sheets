Medallion Architecture (Bronze / Silver / Gold)

Medallion Architecture organizes data into three progressive quality layers inside a Lakehouse. Each layer refines the data further. Coined by Databricks. Also known as Multi-Hop Architecture.

🏗

Layers

1

Bronze (Raw Layer)

Purpose

Land raw data exactly as received from source. Immutable system of record.

Format

Parquet or Delta, original schema preserved

2

Silver (Cleansed / Conformed Layer)

Purpose

Cleansed, deduplicated, validated, and standardized data. Enterprise-wide single source of truth.

Format

Delta Lake / Iceberg with enforced schema

3

Gold (Business / Aggregated Layer)

Purpose

Business-ready, aggregated, enriched data optimized for specific use cases (BI, ML, reporting).

Format

Delta Lake / Parquet, columnar, partitioned

📌

Full Data Flow

Pipeline["1. Kafka event → Bronze Delta table (append raw JSON)","2. Spark Structured St…
⚠️

Drawbacks

  • •Storage cost multiplied across three layers
  • •Pipeline latency compounded at each hop
  • •Managing schema changes across all three layers is complex
  • •Gold layer can proliferate with too many domain-specific tables
✅

Solutions

  • ✓Use Delta VACUUM to clean up old Bronze files
  • ✓Implement Delta Live Tables (DLT) for automated pipeline management
  • ✓Use data contracts to govern Gold table schemas
  • ✓Limit Gold tables to well-defined domains

Data Mesh

Data Mesh is a decentralized sociotechnical architecture where data ownership is distributed to domain teams rather than a central data team. Each domain treats its data as a product. Introduced by Zhamak Dehghani (2019).

📌

Four Principles

1

1. Domain-Oriented Decentralized Data Ownership

Each business domain (orders, payments, inventory) owns, produces, and serves its own data. No central data team owns everything.

Example

The Orders team owns the orders data product. They manage its pipeline, quality, and schema — not a central DE team.

2

2. Data as a Product

Each domain's data must be treated as a product with SLAs, documentation, discoverability, and quality guarantees — not just a by-product of operations.

3

3. Self-Serve Data Infrastructure Platform

A central platform team provides the infrastructure tooling (storage, compute, cataloging, lineage) that domain teams use autonomously. They don't build pipelines — they build the platform.

4

4. Federated Computational Governance

Governance policies (PII tagging, SLAs, retention) are defined centrally but enforced automatically via the platform — not manually by a central team.

Example

Platform auto-tags columns named 'email' or 'ssn' as PII. Masking applied automatically at query time.

💡

Example

CompanyLarge e-commerce platform
Domains[{"domain":"Orders Domain","data_products":["orders_stream","order_history_table…
Platform TeamProvides S3, Glue catalog, Databricks workspace, DataHub, monitoring
⚠️

Drawbacks

  • •Requires significant organizational change — not just a technical solution
  • •Domain teams need data engineering skills (federated DE knowledge)
  • •Risk of duplicated tooling and effort across domains
  • •Federated governance is hard to enforce consistently
  • •Discoverability across many data products requires excellent cataloging
  • •Interoperability between domain data products needs careful standards
✅

Solutions

  • ✓Strong platform team to remove toil for domain teams
  • ✓Standardized data product templates and CI/CD
  • ✓Automated governance via policy-as-code
  • ✓Centralized data catalog with domain-level ownership
  • ✓Data contracts between producer and consumer domains

Data Fabric

Data Fabric is an architecture that provides a unified, intelligent data management layer across heterogeneous, distributed environments. Unlike Data Mesh (organizational), Data Fabric is primarily a technology-driven approach using AI/ML to automate data integration, governance, and access.

✅

Key Capabilities

  • ✓Active metadata — AI learns from how data is used to make recommendations
  • ✓Knowledge graph — maps relationships between all data assets
  • ✓Automated data integration across clouds, on-premise, and edge
  • ✓Unified access layer — one endpoint to query data anywhere
  • ✓Automated governance — policies applied across all data
📌

Data Mesh Vs Data Fabric

Data MeshOrganizational paradigm — decentralize ownership to domains
Data FabricTechnology paradigm — unify access across distributed systems via intelligent automation
Key DifferenceThey are complementary. Data Mesh tells you WHO owns data. Data Fabric tells you HOW to connect and access it.
Can Be Combinedtrue

Tools

Informatica IDMCIBM Cloud Pak for DataTalend Data FabricDenodo (data virtualization)
⚠️

Drawbacks

  • •Heavy vendor dependency
  • •High cost of enterprise fabric platforms
  • •AI-driven recommendations require large volumes of metadata to be useful
  • •Complex to implement across truly heterogeneous environments

Data Hub

A Data Hub is a centralized integration point that collects, harmonizes, and distributes data between systems. It acts as an operational integration layer rather than an analytical store. Manages master data and operational data flows.

📌

Difference From Warehouse

Data Warehouse = analytical queries on historical data. Data Hub = operational integration, master data management, real-time data distribution.

🔩

Components

  • Master Data Management (MDM) — golden record for customers, products
  • Data Integration — connectors to source systems
  • Data Distribution — publish data to consuming systems
  • Metadata Management — catalog of what data exists
  • Data Quality — validation at integration point
💡

Example

Use CaseCustomer 360 Data Hub
Sources["CRM (Salesforce)","E-commerce platform","Support system","Billing system"]…
Hub OutputGolden customer record with merged, deduplicated attributes
Consumers["Analytics warehouse","Marketing automation","Support portal","Mobile app"]…
⚠️

Drawbacks

  • •Hub becomes a critical single point of failure
  • •Tight coupling — all systems depend on the hub
  • •Can become a bottleneck at high throughput

Event-Driven Architecture

Event-Driven Architecture (EDA) is a design paradigm where components communicate by producing and consuming events asynchronously. Systems are decoupled — producers don't know about consumers.

📌

Core Concepts

1

Event

An immutable record of something that happened. Contains: event type, timestamp, payload, source.

Example

{"event_type": "order_placed", "timestamp": "2024-01-15T14:30:00Z", "order_id": "ORD-123", "customer_id": "C-456", "amount": 99.99}

2

Event Producer

Publishes events when something happens in the business domain.

Example

Order service publishes 'order_placed' event when customer completes checkout

3

Event Broker

Routes, stores, and delivers events between producers and consumers.

4

Event Consumer

Subscribes to events and reacts. Multiple consumers can independently process the same event.

Example

Inventory service, notification service, analytics pipeline all consume 'order_placed' independently

💡

Example

Use CaseOrder processing pipeline
Flow["Customer places order → Order Service emits 'order_placed' to Kafka","Inventor…
BenefitAll 5 consumers operate independently. Adding a 6th consumer requires zero changes to the Order Service.
📌

Patterns

1

Event Sourcing

Store all state changes as events. Rebuild state by replaying events.

2

CQRS

Command Query Responsibility Segregation — separate write model from read model.

3

Saga Pattern

Manage distributed transactions via a sequence of events and compensating actions.

⚠️

Drawbacks

  • •Eventual consistency — consumers may lag behind producers
  • •Debugging asynchronous flows is complex
  • •Event schema changes break consumers if not managed carefully
  • •Ordering guarantees across partitions are complex
  • •Duplicate event delivery requires idempotent consumers
✅

Solutions

  • ✓Schema Registry (Confluent) to manage and version event schemas
  • ✓Distributed tracing (OpenTelemetry) for debugging event flows
  • ✓Idempotency keys to handle duplicate processing
  • ✓Dead letter queues for failed event processing

Microservices Data Architecture

In Microservices Architecture, each service owns its own database (Database-per-Service pattern). No shared databases. Services communicate via APIs or events. This creates data integration challenges for analytics.

📌

Database Per Service Pattern

DescriptionEach microservice has its own private database. No other service can directly access it.
Example{"order_service":"PostgreSQL — orders, order_items","inventory_service":"MongoDB…
📌

Analytics Challenge

How do you run reports across data owned by different services with different databases?

📌

Solutions For Analytics

1

Event Streaming to Data Lake

Each service emits events to Kafka. Data platform consumes all events into data lake for analytics.

Flow

Services → Kafka → Kafka Connect → S3/Delta Lake → dbt → Warehouse → BI

2

CDC per Service

Debezium captures changes from each service's DB and streams to central analytics store.

Flow

Service DB → Debezium CDC → Kafka → Analytics Lake

3

API Aggregation

Analytics service calls APIs of each microservice to pull data. Slow but simple.

Drawback

High API load, tight coupling, slow for large datasets

⚠️

Drawbacks

  • •Distributed transactions are very complex (no cross-service ACID)
  • •Data consistency across services is eventual
  • •Joining data across services requires event streaming or API calls
  • •Analytics teams must understand many different data models
