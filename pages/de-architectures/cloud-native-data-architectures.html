Cloud-Native Data Architectures

4 topics

Separation of Storage and Compute

Modern cloud data platforms decouple storage (S3/GCS/ADLS) from compute (Spark clusters, Snowflake warehouses). Scale each independently. Pay only for compute when querying.

✅

Benefits

  • ✓Scale compute up for heavy workloads, scale down to zero when idle
  • ✓Multiple compute engines can query the same data (Spark + Presto + Flink on same S3 data)
  • ✓Storage is cheap and infinitely scalable (S3)
  • ✓No data movement between storage and compute tiers
💡

Example

ArchitectureS3 Data Lake + Databricks
StorageAll data in S3 (Delta format) — $0.023/GB/month
ComputeDatabricks clusters auto-scale: 0 workers (idle) → 20 workers (peak) → 0 workers
Cost SavingPay for compute only during job execution. No always-on cluster.
⚠️

Drawbacks

  • •Network latency between storage and compute (vs local disk)
  • •Cloud egress charges for cross-region data access
  • •Multiple engines on same data require careful concurrency management

Serverless Data Architecture

No servers to provision or manage. Cloud provider auto-scales compute based on demand. Pay per query/execution.

📌

Examples

1

AWS Athena

Serverless SQL on S3. Pay per TB scanned.

2

BigQuery

Serverless warehouse. Pay per TB processed.

3

AWS Glue

Serverless Spark. Pay per DPU-hour.

4

Azure Data Factory

Serverless ETL pipelines.

💡

Example

Use CaseAd-hoc query on S3 data lake with Athena
SetupNo cluster to start. Define table in Glue Catalog pointing to S3 Parquet.
QuerySELECT region, SUM(amount) FROM s3_orders WHERE year=2024 GROUP BY region;
CostScans 500MB Parquet → $0.0025 per query
Startup Time< 2 seconds (no cluster warm-up)
⚠️

Drawbacks

  • •Cold start latency for infrequent workloads
  • •Cost unpredictable for large scan queries
  • •Less control over performance tuning (no cluster config)
  • •Not suitable for long-running complex ETL jobs

Multi-Cloud Architecture

Data platform spans multiple cloud providers (AWS + GCP + Azure). Avoids vendor lock-in. Uses best-of-breed services across clouds.

🎯

Use Cases

  • →Business acquired a company on different cloud
  • →Regulatory requirement to store data in specific regions (different cloud per region)
  • →Best service selection: GCP Vertex AI + AWS S3 + Azure AD
⚠️

Challenges

  • •Data gravity — moving data between clouds incurs egress costs
  • •Security and networking complexity across clouds
  • •Different IAM models per cloud
  • •Latency for cross-cloud data access
✅

Solutions

  • ✓Open table formats (Iceberg) enable any engine on any cloud to query same data
  • ✓Snowflake / Databricks as cross-cloud abstraction layer
  • ✓Keep most data in primary cloud; replicate only necessary subsets

Shared Nothing Architecture

Each processing node has its own CPU, memory, and storage. No shared resources between nodes. Enables massive parallel processing (MPP). Used by Redshift, Snowflake, Teradata.

📌

How It Works

Leader NodeReceives queries, generates execution plan, distributes to compute nodes
Compute NodesEach node processes its own data slice in parallel
DistributionData distributed across nodes by distribution key (hash or round-robin)
Result AggregationLeader node collects results from all compute nodes and returns final result
💡

Example

Redshift Cluster1 leader + 8 compute nodes
QuerySELECT region, SUM(amount) FROM orders GROUP BY region
ExecutionEach compute node sums its own orders partition → leader aggregates 8 partial sums → returns result
Speedup8x faster than single node (ideal case)
⚠️

Drawbacks

  • •Data skew: if distribution key is poorly chosen, some nodes get much more data
  • •Cross-node data movement (broadcasting/shuffling) is expensive
  • •Fixed cluster size — scaling requires redistributing data
