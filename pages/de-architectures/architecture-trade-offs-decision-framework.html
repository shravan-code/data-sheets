Consistency vs Availability Trade-offs

🧭

Decision Framework

1

Financial transactions (payments, billing)

Choose

Consistency (CP)

Reason

Wrong balance is worse than temporary unavailability

2

User social feed, shopping cart

Choose

Availability (AP)

Reason

Showing slightly stale data is better than error

3

Inventory management

Choose

CP for writes, AP for reads

Reason

Must be accurate for writes, ok to show slightly stale stock count

Latency vs Throughput Trade-offs

Optimizing for low latency (fast individual response) often conflicts with optimizing for high throughput (many requests per second).

📌

Examples

1

Batch vs streaming

Streaming (ms)

Throughput

Micro-batching frequency

Latency

Smaller batch = lower latency

Throughput

Smaller batch = more overhead = lower throughput

acks=1 is faster

Compression

Choosing the Right Architecture

🌳

Decision Tree

1

What is your latency requirement?

If Seconds To Hours

Batch or micro-batch pipeline

If Milliseconds

Streaming architecture (Kafka + Flink)

2

Data types?

If Structured Only

Data Warehouse (Snowflake/Redshift/BigQuery)

If Mixed

Data Lakehouse (Delta/Iceberg)

3

Team size and skills?

If Small Sql Heavy

Modern Data Stack (Fivetran + Snowflake + dbt)

If Large Engineering

Custom lakehouse on Databricks or open-source

4

Domain ownership model?

If Centralized

Medallion architecture with central DE team

If Federated

Data Mesh with domain-owned data products

5

Budget?

If Limited

Open-source stack (Spark + Iceberg + Airflow on EC2)

If Flexible

Managed SaaS (Databricks + dbt Cloud + Fivetran)

Common Anti-Patterns

🚫

Anti Patterns

1

Data Swamp

Data lake with no governance, no catalog, no lineage — data exists but nobody knows what it means or trusts it

Solution

Implement data catalog, lineage, data contracts, and data quality checks from day 1

2

Too Many Point-to-Point Pipelines

Every team builds direct pipelines to every other team's data. N*(N-1) pipelines = spaghetti

Solution

Central event bus (Kafka) or data platform layer. All producers publish to platform; consumers read from platform

3

Over-Engineering for Day 1

Building Data Mesh microservices for a 5-person startup with 1GB data

Solution

Start simple (Postgres + dbt + Metabase). Scale architecture as data volume and team grow

4

Missing Idempotency

Non-idempotent pipelines create duplicate data on reruns after failures

Solution

UPSERT patterns, deterministic write paths, watermark-based incremental loads

5

No Schema Management

Source system changes column name → all downstream pipelines silently break or produce wrong results

Solution

Schema Registry, data contracts, schema evolution testing in CI/CD

6

Ignoring Data Skew

Hash partition on a skewed key → 90% data in 1 partition → 1 slow task holds up entire job

Solution

Salt keys, repartition, use salted joins, choose high-cardinality partition keys
