Lambda Architecture

Lambda Architecture handles massive data volumes by combining batch and streaming processing in parallel. Coined by Nathan Marz (2011). Serves as the foundation for understanding modern DE architectures.

🏗

Layers

1

Batch Layer

Responsibility

Stores the master dataset (immutable, append-only). Periodically recomputes batch views from scratch.

Latency

Hours

Hadoop HDFSApache SparkHive
2

Speed Layer (Stream Layer)

Responsibility

Processes real-time data to fill the latency gap of the batch layer. Views are approximate and temporary.

Latency

Milliseconds to seconds

Apache StormSpark StreamingKafka Streams
3

Serving Layer

Responsibility

Merges batch views and real-time views to answer queries. Responds to ad-hoc queries.

Latency

Low-latency reads

HBaseCassandraElasticsearchDruid
📌

⚠️

Drawbacks

  • •Two separate codebases for batch and streaming logic — must be kept in sync
  • •High operational complexity — running two processing systems
  • •Batch layer latency means stale data until next batch run
  • •Reprocessing the full dataset is expensive and slow
  • •Debugging across two systems is difficult
📌

Solutions Evolution

These drawbacks led to Kappa Architecture (stream-only) and eventually Lakehouse architecture

Kappa Architecture

Kappa Architecture (Jay Kreps, 2014) simplifies Lambda by eliminating the batch layer entirely. Everything is treated as a stream. Historical reprocessing is done by replaying events from the log (Kafka retention).

📌

Core Principle

A stream processing system that can replay historical data IS a batch system. No need to maintain two separate systems.

🏗

Layers

1

Stream Layer (Single Layer)

Responsibility

All data — historical and real-time — is processed as a stream. Kafka retains full event history. To reprocess, reset the consumer offset to the beginning.

Apache KafkaApache FlinkKafka StreamsApache Samza
2

Serving Layer

Responsibility

Stores the output of stream processing for query serving.

CassandraElasticsearchRedisPostgreSQL

Example

Use CaseReal-time user behavior analytics
SetupKafka topic 'user_events' retains 1 year of data
Normal FlowFlink reads new events → computes per-user session stats → writes to Cassandra
ReprocessingNew feature: add geo-enrichment. Deploy new Flink job → reset offset to Jan 1 → replay all events → recompute with geo-enrichment → overwrite serving layer
⚠️

Drawbacks

  • •Long reprocessing times for very large historical datasets
  • •Kafka storage costs increase with long retention periods
  • •Streaming logic is more complex than batch SQL (state management, ordering)
  • •Not ideal for complex aggregations that are naturally batch-oriented
  • •Backfilling years of data through a stream system can be slow
📌

Solutions Evolution

Lakehouse architecture addresses these by providing efficient batch queries on the same data store
