Fan-Out Pattern

A single event or message is distributed to multiple downstream consumers or processing branches simultaneously. Decouples producers from multiple consumers.

💡

Example

Use CaseOrder placed event → 5 downstream systems
Diagram{"source":"Kafka: orders.placed","consumers":["Inventory Service: reserve stock"…
ImplementationSingle Kafka topic with 5 independent consumer groups. Each group reads independently.
⚠️

Drawbacks

  • •One slow consumer doesn't block others but creates consumer lag
  • •Managing 5 independent pipelines increases operational overhead

Fan-In Pattern

Multiple input streams or sources are merged into a single output stream. Useful for aggregating events from many sources.

💡

Example

Use CaseUnified audit log from multiple microservices
Sources["orders-events topic","payments-events topic","inventory-events topic"]…
Outputunified-audit-events topic
Flink CodeDataStream<Event> merged = ordersStream .union(paymentsStream) .union(inventoryStream);

Saga Pattern

Manages long-running distributed transactions across multiple services using a sequence of local transactions. If a step fails, compensating transactions undo previous steps.

📌

Types

1

Choreography Saga

Each service publishes events and listens for events from other services. No central coordinator.

Example

Order Service publishes order_created → Payment Service listens, charges card, publishes payment_completed → Inventory Service listens, reserves stock

Drawback

Hard to track overall transaction state

2

Orchestration Saga

A central Saga Orchestrator tells each service what to do and handles failures.

Example

Orchestrator: 1. Call Payment Service → 2. Call Inventory Service → 3. Call Shipping Service. If step 2 fails → call Payment Service compensate endpoint.

Drawback

Orchestrator becomes a single point of failure

📌

Compensating Transaction Example

Happy Path["Reserve inventory","Charge payment","Create shipment"]…
Failure At Step 3["Undo: Refund payment","Undo: Release inventory reservation"]…

Outbox Pattern

Guarantees that database writes and event publishing happen atomically. Solves the dual-write problem: how to write to DB AND publish to Kafka without risk of one succeeding and the other failing.

📌

Dual Write Problem

Without outbox: write to DB succeeds, Kafka publish fails → DB has data, Kafka missing event → inconsistency

📌

Outbox Solution

Step 1Write to application DB table AND outbox table in same transaction
Step 2Outbox relay (Debezium CDC) reads outbox table, publishes to Kafka
Step 3Delete/mark outbox row after successful publish
GuaranteeBoth DB write and Kafka publish are atomic — no dual-write inconsistency
📌

Outbox Table

Code
CREATE TABLE outbox (
  id UUID PRIMARY KEY,
  event_type VARCHAR,
  payload JSONB,
  created_at TIMESTAMP,
  published_at TIMESTAMP
);

Circuit Breaker Pattern

Prevents a pipeline from repeatedly calling a failing downstream service, giving it time to recover. Like an electrical circuit breaker — trips when too many failures occur.

📌

States

1

Closed (normal)

Calls pass through. Failure count tracked.

2

Open (tripped)

Calls blocked immediately. Fallback used. Wait for reset timeout.

3

Half-Open

Allow limited test calls. If successful → close. If fail → reopen.

💡

Example

ScenarioPipeline writing to downstream API
ConfigTrip after 5 failures in 30s. Reset after 60s.
Behavior5 API timeouts in 30s → circuit opens → pipeline uses cached fallback response for 60s → half-open: 1 test call → success → circuit closes

Retry & Backoff Pattern

Automatically retry failed operations with increasing delays to avoid hammering a recovering service.

📌

Backoff Strategies

1

Fixed Retry

Retry every N seconds

Example

Retry every 5 seconds, max 3 times

2

Exponential Backoff

Double the wait time after each failure

Example

Retry after 1s, 2s, 4s, 8s, 16s

3

Exponential Backoff + Jitter

Add random jitter to avoid thundering herd (all retries hitting service at same time)

Formula

wait = min(cap, base * 2^attempt) + random_jitter

Example

Retry after 1s±0.5, 2s±1, 4s±2, 8s±4

📌

Python Example

Code
import time, random
def retry_with_backoff(fn, max_retries=5):
    for attempt in range(max_retries):
        try:
            return fn()
        except Exception as e:
            if attempt == max_retries - 1:
                raise
            wait = min(60, (2 ** attempt)) + random.uniform(0, 1)
            time.sleep(wait)

Idempotency Pattern

An operation is idempotent if applying it multiple times produces the same result as applying it once. Critical for safe retries in distributed pipelines.

📌

Examples

1

Database write

Non Idempotent

INSERT INTO orders VALUES (...) — duplicate rows on retry

Idempotent

INSERT INTO orders VALUES (...) ON CONFLICT (order_id) DO UPDATE SET ... — safe to retry

2

S3 write

Non Idempotent

Append to file — duplicates on retry

Idempotent

Write to deterministic path (date-keyed) — overwrite on retry

3

API call

Non Idempotent

POST /charge — duplicate charges on retry

Idempotent

PUT /charge/{idempotency_key} — server deduplicates by key
