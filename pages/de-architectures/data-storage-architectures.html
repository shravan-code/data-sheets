Data Storage Architectures

6 topics

Data Warehouse

A Data Warehouse is a centralized, structured repository optimized for analytical queries (OLAP). Data is cleaned, transformed, and modeled (star/snowflake schema) before loading. Schema-on-write approach.

⚡

Characteristics

  • Structured data only (tables, schemas)
  • Schema-on-write (schema defined before loading)
  • Highly optimized for SQL reads and aggregations
  • Columnar storage for analytical performance
  • Supports ACID transactions
  • MPP (Massively Parallel Processing) architecture

  • •Schema must be defined upfront — inflexible for exploratory analysis
  • •ETL pipelines are time-consuming to build and maintain

Data Lake

A Data Lake is a centralized repository that stores all data in its raw, native format — structured, semi-structured, and unstructured. Schema-on-read: schema is applied only at query time.

⚡

Characteristics

  • Stores all data types: CSV, JSON, Parquet, images, video, logs
  • Schema-on-read (no upfront schema required)
  • Cheap storage (object storage: S3, GCS, ADLS)
  • Supports batch, streaming, ML, and ad-hoc analytics
  • Decoupled storage and compute
  • No ACID transactions (raw files)
📌

Zones

1

Raw / Landing Zone

Exact copy of source data, never modified. Acts as system of record.

Format

Original format (CSV, JSON, Avro)

2

Cleansed / Standardized Zone

Data cleaned, deduplicated, standardized schema.

Format

Parquet / ORC

3

Curated / Enriched Zone

Business-ready, enriched, joined datasets.

Format

Parquet / Delta

  • •Easily becomes a 'data swamp' without governance
  • •No ACID transactions — no updates/deletes on files
  • •Poor query performance on raw files without optimization
  • •Schema drift and inconsistent formats over time
  • •Hard to enforce data quality and lineage
  • •Security and access control complex at file level

Data Lakehouse

Data Lakehouse combines the low-cost open storage of Data Lakes with the reliability, ACID transactions, and performance of Data Warehouses. Achieved through open table formats (Delta Lake, Apache Iceberg, Apache Hudi) on top of object storage.

⚡

Characteristics

  • ACID transactions on object storage (S3/GCS/ADLS)
  • Schema enforcement AND schema evolution
  • Time travel (query historical snapshots)
  • Open formats (Parquet + metadata layer)
  • Unified batch and streaming
  • Support for BI, ML, and streaming on same data
  • Decoupled storage and compute
🏛

Architecture

Storage LayerObject storage (S3/GCS/ADLS) — cheap, scalable
Table Format LayerDelta Lake / Iceberg / Hudi — adds ACID, versioning, metadata
Compute LayerSpark, Flink, Trino, Presto, DuckDB — query engines
Catalog LayerHive Metastore, AWS Glue, Unity Catalog — metadata management

Data Mart

A Data Mart is a subset of a Data Warehouse focused on a single business domain (sales, finance, HR). Smaller, faster, and tailored to specific user groups.

📌

Types

1

Dependent Data Mart

Sourced from the central data warehouse. Consistent, governed.

Diagram

Data Warehouse → ETL → Data Mart

2

Independent Data Mart

Sourced directly from operational systems, bypassing the warehouse. Faster to build but risky — inconsistent definitions.

Diagram

Operational DB → ETL → Data Mart

3

Hybrid Data Mart

Combines both: primarily from warehouse but supplements with direct source feeds.

Diagram

Data Warehouse + Operational DB → Data Mart

💡

Tables["fact_revenue","fact_expenses","dim_cost_center","dim_account","dim_date"]…

⚠️

Drawbacks

  • •Data mart proliferation — each team builds its own, causing inconsistencies
  • •Independent marts create conflicting metrics ('which revenue number is right?')
  • •Maintenance overhead of multiple marts

Operational Data Store (ODS)

An ODS is a database that integrates data from multiple operational sources for current (or near-current) operational reporting. Unlike a warehouse, it stores current data with frequent updates rather than historical aggregates.

⚡

Characteristics

  • Near-real-time data (minutes delay vs hours for warehouse)
  • Normalized or lightly denormalized schema
  • Supports operational queries and reporting
  • Data refreshed frequently (every few minutes)
  • Acts as integration hub between OLTP and OLAP
📌

Data Flow

OLTP Systems → CDC/ETL → ODS → (periodic) → Data Warehouse

💡

Example

Use CaseCustomer service ODS
Sources["CRM system","Order management system","Support ticketing system"]…
Ods Tables["customer_current_state","open_orders","active_support_tickets"]…
QuerySELECT c.name, o.order_status, t.ticket_status FROM customer_current_state c JOIN open_orders o ON c.id = o.customer_id JOIN active_support_tickets t ON c.id = t.customer_id WHERE c.id = 12345
⚠️

Drawbacks

  • •Not optimized for complex analytical queries (that's what the warehouse is for)
  • •Data freshness still has some lag vs true real-time
  • •Requires careful capacity planning for write-heavy workloads
