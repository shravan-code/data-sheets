Data Catalog Architecture

A data catalog is a metadata inventory of all data assets — tables, columns, pipelines, dashboards. It enables data discovery, lineage tracking, and governance.

🔩

Components

1

Technical Metadata

Schema, data types, row counts, partition info, last updated

2

Business Metadata

Business definitions, owners, SLAs, data domains

3

Operational Metadata

Pipeline run history, data quality scores, freshness

4

Social Metadata

Column-level descriptions, popularity scores, user ratings

💡

Example

Search ExampleSearch 'customer revenue' → finds gold.customer_revenue_daily table, owner: analytics team, updated: 2 hours ago, upstream: silver.orders + silver.customers, downstream: Tableau Sales Dashboard

Tools

Apache AtlasDataHubCollibraAlationAWS Glue Data CatalogGoogle Data Catalog
⚠️

Drawbacks

  • •Catalog becomes outdated if not auto-populated via pipeline integration
  • •Manual documentation efforts don't scale
  • •Requires organizational buy-in to maintain descriptions and ownership

Data Lineage

Data lineage tracks the origin, movement, and transformation of data — where it came from, what changed it, and where it went. Essential for impact analysis and debugging.

📌

Types

1

Column-Level Lineage

Tracks which source columns contribute to each target column

Example

gold.revenue.total_amount ← silver.orders.amount + silver.orders.tax_amount ← bronze.raw_orders.amount_str (after CAST)

2

Table-Level Lineage

Tracks which tables feed into which tables

Example

bronze.raw_orders → silver.orders → gold.daily_sales

3

Pipeline Lineage

Tracks which pipeline/job produced which dataset

Example

Airflow DAG: daily_etl → Spark job: clean_orders → Delta table: silver.orders

🎯

Use Cases

  • →Impact Analysis: if silver.orders schema changes, which Gold tables and dashboards are affected?
  • →Root Cause: incorrect revenue number → trace back to source transformation bug
  • →GDPR: which tables contain customer_id 12345's data? (for right-to-be-forgotten)

Tools

Apache AtlasDataHubOpenLineagedbt lineage graphMarquez
⚠️

Drawbacks

  • •Automated lineage extraction requires deep integration with all tools
  • •Column-level lineage from complex SQL is hard to parse automatically
  • •Lineage graphs become hard to navigate for large data platforms

Data Observability

Data observability monitors the health of data across five pillars. Like application monitoring (Datadog) but for data pipelines and datasets.

📌

Five Pillars

1

Freshness

Is data being updated on schedule?

Alert

orders table not updated in 3 hours (SLA: 1 hour)

2

Volume

Is the expected amount of data arriving?

Alert

orders table received 0 rows today (expected 50K+)

3

Distribution

Are column value distributions normal?

Alert

order_amount mean dropped from $85 to $2 — possible data error

4

Schema

Has schema changed unexpectedly?

Alert

Column 'customer_id' dropped from silver.orders

5

Lineage

Are upstream dependencies healthy?

Alert

bronze.raw_orders (upstream) failed — silver.orders may be stale

Tools

Monte CarloSoda CoreGreat Expectationsdbt testsAWS DeequBigeye
⚠️

Drawbacks

  • •False positives from legitimate business events (holiday spikes, promotions)
  • •Statistical anomaly detection requires tuning baseline windows
  • •Tool cost scales with data volume

Data Quality Framework

A systematic approach to measuring, monitoring, and enforcing data quality across pipelines.

📌

Quality Dimensions

1

Completeness

Check

NULL rate < 1% for critical columns

Example

order_id IS NOT NULL

2

Accuracy

Check

Values conform to business rules

Example

amount > 0 AND amount < 1000000

3

Consistency

Check

Same entity has same values across tables

Example

customer_id in orders exists in customers table

4

Timeliness

Check

Data arrives within SLA

Example

MAX(updated_at) > NOW() - INTERVAL '2 hours'

5

Uniqueness

Check

No duplicate records on key columns

Example

COUNT(*) = COUNT(DISTINCT order_id)

6

Validity

Check

Values are within expected domain

Example

status IN ('pending','processing','shipped','delivered','returned')

📌

Great Expectations Example

Codefrom great_expectations.dataset import PandasDataset ds = PandasDataset(orders_df) ds.expect_column_values_to_not_be_null('order_id') ds.expect_column_values_to_be_in_set('status', ['pending','shipped','delivered']) ds.expect_column_values_to_be_between('amount', 0.01, 100000) results = ds.validate()

Data Contracts

A data contract is a formal, machine-readable agreement between a data producer and consumer defining the schema, semantics, SLAs, and quality guarantees of a dataset.

📌

Contract Components

  • Schema: column names, data types, nullable constraints
  • Semantics: what each column means (business definitions)
  • SLAs: freshness, availability, completeness guarantees
  • Quality: expected value ranges, distributions
  • Versioning: how breaking changes are communicated and handled
  • Owner: who is responsible for this data product
📌

Schema Registry

A centralized repository that stores and manages schema versions for event data (Avro, Protobuf, JSON Schema). Ensures producers and consumers use compatible schemas.

📌

Flow

Producer1. Producer registers schema with registry → gets schema_id 2. Producer serializes event: [magic_byte][schema_id][avro_payload] 3. Sends to Kafka
Consumer1. Consumer reads [schema_id] from message 2. Fetches schema from registry 3. Deserializes payload using schema
📌

Compatibility Modes

1

BACKWARD

New schema can read data written by old schema (add optional fields)

2

FORWARD

Old schema can read data written by new schema

3

FULL

Both backward and forward compatible

4

NONE

No compatibility check — dangerous

Tools

Confluent Schema RegistryAWS Glue Schema RegistryApicurio
⚠️

Drawbacks

  • •Schema Registry becomes critical infrastructure — must be highly available
  • •Breaking schema changes require all consumers to update
  • •Complex to manage across many teams and schemas

PII & Data Masking Architecture

PII (Personally Identifiable Information) must be detected, tagged, and masked/encrypted to comply with GDPR, CCPA, HIPAA. Architecture must enforce PII handling across all layers.

📌

Pii Detection

AutomatedScan column names and values for PII patterns: email, SSN, phone, name, address, IP
Tools["AWS Macie","Google DLP","Presidio (open-source)"]…
📌

Masking Techniques

1

Redaction

Example

alice@email.com → ****@***.***

Logs, debugging
2

Tokenization

Example

4242-4242-4242-4242 → TKN-8472-XA

Preserve referential integrity
3

Hashing

Example

alice@email.com → SHA256 hash

Join on anonymized ID
4

Generalization

Example

Age 34 → Age 30-35

Analytics on anonymized data
5

Encryption

Example

AES-256 encryption

Store encrypted, decrypt for authorized users
📌

Architecture Layers

IngestionDetect and tag PII columns via automated scanner
StorageStore PII in encrypted columns; tag in data catalog
ProcessingApply column masking at query time for non-privileged users
ServingRow-level and column-level security in warehouse
