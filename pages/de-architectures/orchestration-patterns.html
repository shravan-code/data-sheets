Orchestration Patterns

5 topics

DAG-Based Orchestration

Directed Acyclic Graph (DAG) — tasks represented as nodes, dependencies as directed edges. The orchestrator executes tasks in topological order, respecting dependencies.

💡

Example

ToolApache Airflow
Dag{"name":"daily_sales_pipeline","schedule":"0 2 * * *","tasks":{"extract_orders":…
Parallel Executionextract_orders and extract_customers run in parallel since neither depends on the other
⚠️

Drawbacks

  • •Static DAGs require code deployment for every change
  • •Cron-based scheduling is time-based, not data-arrival-based
  • •Airflow scheduler can become a bottleneck at scale

Event-Driven Orchestration

Pipeline tasks are triggered by events (data arrival, upstream completion, external signals) rather than time-based schedules. More reactive and efficient.

💡

Example

ScenarioS3 file arrival triggers pipeline
Flow["Vendor uploads file to S3","S3 event notification → Lambda → triggers Airflow …
Tools["Airflow Sensors","AWS EventBridge + Step Functions","Dagster asset-based sched…
⚠️

Drawbacks

  • •Harder to reason about than time-based schedules
  • •Event loss can silently skip pipeline runs
  • •Debugging event-triggered flows requires event tracing

Choreography vs Orchestration

Two models for coordinating distributed processes.

📊

Comparison Table

Featureorchestrationchoreography
ControlCentral orchestrator controls flowEach service reacts to events independently
VisibilityEasy — orchestrator knows full stateHard — flow is implicit in event chain
CouplingServices depend on orchestratorServices loosely coupled via events
Failure handlingOrchestrator handles rollbackEach service handles own compensation
Best forComplex multi-step pipelines with clear ownerSimple event chains, highly decoupled teams

Dynamic DAGs

DAGs generated programmatically at runtime based on config, database state, or external parameters. Enables parameterized and data-driven pipeline generation.

💡

Example

Use CaseGenerate one DAG per active client
Airflow Codeimport yaml clients = yaml.safe_load(open('clients.yaml')) for client in clients: dag_id = f'ingest_{client["name"]}' dag = DAG(dag_id, schedule_interval=client['schedule']) with dag: extract = PythonOperator(task_id='extract', python_callable=extract_fn, op_kwargs={'client': client}) load = PythonOperator(task_id='load', python_callable=load_fn, op_kwargs={'client': client}) extract >> load globals()[dag_id] = dag
Result50 clients → 50 auto-generated DAGs
⚠️

Drawbacks

  • •Dynamic DAGs make the Airflow UI noisy with many DAGs
  • •Hard to debug when generated DAG has issues
  • •Airflow parses all DAG files every heartbeat — too many dynamic DAGs can slow scheduler

SLA & Alerting Patterns

SLAs define expected completion times for pipelines. Alerting notifies teams when SLAs are breached or tasks fail.

📌

Sla Types

1

Data Freshness SLA

Example

Sales dashboard must show data no older than 2 hours

2

Pipeline Completion SLA

Example

Daily ETL must complete by 6am

3

Data Quality SLA

Example

NULL rate for order_id must be < 0.01%

📌

Airflow Sla Example

DagDAG('daily_etl', sla_miss_callback=sla_alert_fn, default_args={'sla': timedelta(hours=4)})
Alert Functiondef sla_alert_fn(dag, task_list, blocking_task_list, slas, blocking_tis): send_slack_alert(f'SLA missed: {task_list}')
📌

Alerting Channels

  • Slack/Teams webhook
  • PagerDuty on-call
  • Email
  • Datadog/Grafana alerts
