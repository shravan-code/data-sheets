What is the difference between Lambda and Kappa architecture?

Answer

Lambda uses two separate processing pipelines: a batch layer for accuracy on historical data and a speed layer for real-time results. Kappa eliminates the batch layer — everything is a stream, with historical reprocessing done by replaying the event log from the beginning. Lambda's main drawback is maintaining two codebases with the same logic. Kappa solves this with a single streaming codebase but struggles with very large historical reprocessing.

⚠️Lambda Drawback

Duplicate logic in batch and stream code must be kept in sync

⚠️Kappa Drawback

Replaying years of data through streaming can be very slow

✅Modern Solution

Data Lakehouse (Delta Lake) + unified batch-stream engines (Flink, Spark Structured Streaming) effectively supersede both

2

What is a Data Lakehouse and how does it differ from a Data Lake?

Answer

A Data Lake stores raw data (any format) on object storage but lacks ACID guarantees, schema enforcement, and efficient querying. A Data Lakehouse adds a table format layer (Delta Lake, Iceberg, Hudi) on top of the data lake, providing ACID transactions, time travel, schema evolution, and efficient query performance — while retaining the cost and flexibility benefits of the lake.

⚠️Drawback

Added complexity of table format metadata, compaction, and vacuum management

✅Solution

Use managed platforms (Databricks) that automate compaction, vacuuming, and Delta Live Tables

3

Explain the Medallion Architecture (Bronze/Silver/Gold).

Answer

Medallion Architecture has three progressive data quality layers. Bronze: raw, immutable source data — exact copy, no transformation. Silver: cleansed, deduplicated, standardized — single source of truth for row-level data. Gold: business-ready aggregates, domain-specific, optimized for analytics and ML. Each layer builds on the previous, improving data quality.

⚠️Drawback

Storage cost multiplied 3x, pipeline latency compounded at each hop

✅Solution

VACUUM old Bronze files, use Delta Live Tables for automated multi-hop management

What is Data Mesh and what are its four principles?

Answer

Data Mesh is a decentralized data architecture where domain teams own their data as products. The four principles are: (1) Domain-oriented decentralized data ownership, (2) Data as a product — each domain's data meets discoverability, quality, and SLA standards, (3) Self-serve data infrastructure platform — central platform team provides tools, not pipelines, (4) Federated computational governance — policies defined centrally, enforced automatically.

⚠️Drawback

Requires major organizational change; domain teams need DE skills; risk of inconsistent tooling

✅Solution

Strong platform team, standardized data product templates, automated governance via policy-as-code, unified data catalog

5

What is CAP theorem and how does it apply to data engineering?

Answer

CAP theorem states a distributed system can only guarantee two of: Consistency (all nodes see same data), Availability (every request gets a response), Partition Tolerance (system works despite network failures). In practice, partition tolerance is mandatory in distributed systems, so the real choice is CP (consistent but may be unavailable during partitions) or AP (always available but may return stale data). In DE: Kafka is AP for delivery guarantees; PostgreSQL in single-node is effectively CA; Cassandra is AP.

⚠️Drawback

PACELC extends CAP to include latency vs consistency trade-off even without partitions

✅Solution

Choose database and consistency model based on business requirement — financial systems need CP, user feeds can tolerate AP

6

What is Change Data Capture (CDC) and how does log-based CDC work?

Answer

CDC captures row-level changes from a source database as they happen. Log-based CDC (Debezium) reads the database's transaction log (PostgreSQL WAL, MySQL binlog) directly — zero query overhead on the source. Each change (INSERT/UPDATE/DELETE) is emitted as an event with before/after values. These events stream to Kafka and then to the data lake.

⚠️Drawback

Requires REPLICATION privilege; schema changes in source can break CDC; transaction log retention must be managed

✅Solution

Schema Registry to handle schema evolution; monitor log retention; use Debezium's snapshot mode for initial load

7

What is the difference between ETL and ELT?

Answer

ETL transforms data outside the destination before loading — data is moved twice (extract → transform → load into warehouse). ELT loads raw data directly into the destination first (cloud warehouse or lake), then transforms it using the warehouse's compute power (dbt SQL). ELT is preferred in modern stacks because raw data is preserved, cloud warehouses have massive compute power, and SQL-based transforms are easier to maintain.

⚠️Elt Drawback

Raw sensitive data (PII) lands directly in warehouse — security risk; warehouse compute costs for heavy transforms

✅Solution

Apply column-level masking at ingestion via Lake Formation or Snowflake Dynamic Data Masking; use incremental dbt models for cost efficiency

8

Explain exactly-once semantics in streaming. How does Flink achieve it?

Answer

Exactly-once means each event is processed exactly once — no duplicates, no data loss. Flink achieves this via distributed checkpointing: periodically saves consistent snapshots of all operator state to durable storage (S3). On failure, Flink restores from the latest checkpoint and replays only events since that checkpoint. Combined with idempotent sinks or Kafka transactions, this guarantees exactly-once end-to-end.

⚠️Drawback

Checkpointing adds latency and throughput overhead; exactly-once requires all components to support it

✅Solution

Tune checkpoint interval based on latency tolerance; for most analytics use cases, at-least-once with idempotent sinks is sufficient and cheaper

9

What is the small file problem and how do you solve it?

Answer

When streaming or micro-batch pipelines write frequently, thousands of tiny files accumulate per partition. Each file requires a separate metadata call to S3/HDFS — 1M small files = 1M metadata calls = very slow queries. Solution: compaction merges small files into fewer large files. Delta Lake OPTIMIZE command; Iceberg rewrite_data_files; Spark repartition before write. Schedule compaction during off-peak hours.

⚠️Drawback

Compaction is compute-intensive and rewrites files — generates new Delta log entries

✅Solution

Run VACUUM after OPTIMIZE to remove obsolete old files; use auto-optimize in Databricks

10

What is data skew in Spark and how do you handle it?

Answer

Data skew occurs when data is unevenly distributed across partitions — one partition has 10x more data than others. The slow task in the large partition blocks the entire stage. Causes: low-cardinality join keys (e.g., NULL values), hot dates, popular customers. Solutions: (1) Salting: add random prefix to skewed key to distribute data, (2) Broadcast join for small tables (avoids shuffle), (3) skewHint in Spark 3, (4) Repartition before join on uniform key.

⚠️Drawback

Salting adds complexity and requires de-salting after aggregation

✅Solution

Spark 3.x Adaptive Query Execution (AQE) automatically detects and handles skew with skew join optimization

11

What is the difference between Apache Iceberg, Delta Lake, and Hudi?

Answer

All three are open table formats providing ACID on data lakes. Delta Lake (Databricks): best Spark integration, OPTIMIZE+ZORDER, unified batch+stream. Apache Iceberg (Netflix): best multi-engine support (Spark+Trino+Flink+Athena), hidden partitioning, richest schema evolution. Apache Hudi (Uber): best for upsert-heavy CDC pipelines, native incremental read API, COW/MOR table types.

⚠️Drawback

Each format has different ecosystem support and operational overhead

✅Solution

Choose based on primary compute engine — Databricks shop: Delta; multi-engine cloud: Iceberg; CDC/upsert heavy: Hudi

12

Explain partitioning strategies for data lake optimization.

Answer

Partitioning divides data into folders by column value, enabling partition pruning (skipping irrelevant partitions on query). Best partition keys: low-to-medium cardinality, frequently used in WHERE filters — date, region, status. Avoid partitioning on high-cardinality columns (user_id) — creates millions of tiny partitions. Use Z-ordering (Delta) for high-cardinality secondary filter columns alongside date partitioning.

⚠️Drawback

Over-partitioning creates small file problem; wrong partition key provides no benefit

✅Solution

Target 100MB-1GB per partition file; combine date partitioning with Z-ordering for compound filter optimization

13

What is a data contract and why is it important?

Answer

A data contract is a formal agreement between a data producer and consumer specifying schema, semantics, SLAs, quality rules, and ownership. It prevents the common problem of source system changes silently breaking downstream consumers. Producers must honor the contract or communicate breaking changes with versioning. Data contracts are the foundation of reliable data products in a Data Mesh.

⚠️Drawback

Hard to enforce contracts without tooling; requires organizational discipline from source teams

✅Solution

Schema Registry for schema validation; automated contract testing in CI/CD; DataHub for contract documentation

14

How does predicate pushdown improve query performance?

Answer

Predicate pushdown moves WHERE clause filters as close to the data source as possible. In Parquet, each row group stores min/max statistics per column. If WHERE amount > 1000 and a row group's max_amount = 500, the entire row group is skipped — zero bytes read. This can reduce I/O by 90%+ for selective queries on large datasets. Z-ordering on Delta Lake amplifies this by co-locating related data.

⚠️Drawback

Functions on columns prevent pushdown (WHERE YEAR(order_date) = 2024)

✅Solution

Rewrite predicates to reference columns directly: WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31'

15

What is the Outbox Pattern and why is it used?

Answer

The Outbox Pattern solves the dual-write problem: atomically publishing to both a database and a message broker. Without it, a DB write can succeed while a Kafka publish fails (or vice versa), creating inconsistency. Solution: write to the application DB AND an outbox table in the same transaction. A CDC relay (Debezium) reads the outbox table and publishes to Kafka, ensuring both writes are atomic.

⚠️Drawback

Outbox table grows if relay is slow; adds operational component

✅Solution

Clean up processed outbox rows regularly; use Debezium's outbox router event transformation

16

What are windowing strategies in stream processing?

Answer

Windows group events for aggregation in streams. Tumbling windows: fixed-size, non-overlapping (every 5 min). Each event belongs to exactly one window. Sliding windows: fixed-size but overlapping (10-min window, 5-min slide). Session windows: dynamic, gap-based — window closes after N minutes of inactivity. Use tumbling for periodic metrics, sliding for rolling averages, session for user behavior analytics.

⚠️Drawback

Out-of-order events can arrive after window closes — data loss or incorrect results

✅Solution

Watermarks define how far behind expected; Allowed Lateness keeps window open for late events; Side Output routes very late events for separate handling

17

Explain the Dead Letter Queue pattern.

Answer

A Dead Letter Queue (DLQ) captures events that fail processing after N retries instead of dropping them or blocking the pipeline. Failed events are routed to a separate DLQ topic with the original payload and error details. Engineers can inspect, fix the root cause, and replay DLQ events. Essential for pipeline reliability — prevents a bad event from blocking all downstream processing.

⚠️Drawback

DLQ can accumulate silently without alerting; replaying DLQ out of order can cause issues

✅Solution

Set up alerts on DLQ depth; implement idempotent processing before replaying; sort DLQ events by timestamp on replay

18

What is the difference between row-level and column-level security?

Answer

Column-level security restricts which columns a user can see — e.g., salary and SSN columns masked for analysts but visible to HR admins. Row-level security restricts which rows a user can see — e.g., EU analysts only see EU region rows. Both can be implemented via dynamic data masking policies (Snowflake), Lake Formation column masking, or database view-based access control.

⚠️Drawback

Performance overhead of evaluating policies on every query; complex policy management at scale

✅Solution

Cache policy results; use role-based policies not user-based policies; test performance impact before deploying

19

What is Schema Evolution and why is it important?

Answer

Schema evolution is the ability to add, remove, or modify table columns over time without breaking existing readers or requiring full table rewrites. Safe changes: add optional column (existing files return NULL), rename column (tracked by column ID not name), widen type (INT→BIGINT). Unsafe: narrow type, add NOT NULL constraint without default. Iceberg and Delta Lake both support schema evolution — Iceberg's support is most comprehensive.

⚠️Drawback

Breaking changes (remove required column) can silently break downstream consumers

✅Solution

Data contracts with versioning; Schema Registry backward compatibility checks; test schema changes in dev before prod

20

How does MVCC (Multi-Version Concurrency Control) work in a Lakehouse?

Answer

In Delta Lake and Iceberg, MVCC is implemented through snapshot isolation. Every commit creates a new snapshot (version). Readers access a specific snapshot — they see a consistent view even while a writer commits changes to a new snapshot. Writers use optimistic concurrency: if two writers try to commit to the same version, one fails with a conflict error and must retry. This enables concurrent reads and writes without blocking.

⚠️Drawback

Old snapshots accumulate — storage grows without cleanup

✅Solution

Regular VACUUM (Delta) or expire_snapshots (Iceberg) with appropriate retention window

21

What is data observability and what are its five pillars?

Answer

Data observability monitors the health of data pipelines and datasets. Five pillars: Freshness (is data updated on schedule?), Volume (expected rows arriving?), Distribution (column value distributions normal?), Schema (unexpected schema changes?), Lineage (are upstream dependencies healthy?). Tools: Monte Carlo, Soda Core, Great Expectations, dbt tests.

⚠️Drawback

False positives from legitimate business events (seasonal spikes); tool cost scales with data volume

✅Solution

Tune anomaly detection baselines to account for seasonality; start with dbt tests (free) before investing in observability SaaS

22

What is Reverse ETL and when would you use it?

Answer

Reverse ETL moves processed, enriched data FROM the warehouse BACK to operational tools — Salesforce, Hubspot, Intercom. Use it when analytics insights should drive business actions: sync customer churn scores to CRM so sales can act, sync product usage metrics to customer success tools, sync personalization segments to marketing platforms.

⚠️Drawback

API rate limits throttle sync speed; identity resolution between warehouse IDs and CRM IDs is complex; risk of overwriting operational data

✅Solution

Use Census/Hightouch which handle rate limiting and ID mapping; implement sync validation to prevent data corruption

23

How do you design an idempotent data pipeline?

Answer

An idempotent pipeline produces the same result regardless of how many times it runs. Techniques: (1) UPSERT instead of INSERT — ON CONFLICT DO UPDATE, (2) Write to deterministic paths — s3://lake/orders/date=2024-01-15/ (overwrite on rerun), (3) DELETE + INSERT for batch partitions, (4) Watermark-based incremental loads with idempotency keys, (5) Delta MERGE for upsert semantics.

⚠️Drawback

UPSERT requires a unique key; deterministic paths require careful partition design

✅Solution

Define business keys early; use composite keys if no single unique key exists; test idempotency by running pipeline 3x and comparing results

24

What is data lineage and how do you implement it?

Answer

Data lineage tracks the lifecycle of data: where it came from, how it was transformed, and where it went. Implementation approaches: (1) OpenLineage standard — Airflow and Spark emit lineage events, (2) dbt automatically generates column-level lineage from SQL models, (3) DataHub/Atlas collect lineage from multiple sources, (4) Marquez as OpenLineage backend.

⚠️Drawback

Column-level lineage from complex Python transforms is hard to capture automatically

✅Solution

Standardize transforms in SQL (dbt) for automatic lineage; manually document complex Python transforms in catalog

25

What is stream-table duality in Kafka?

Answer

In Kafka Streams/ksqlDB, a stream and a table are two views of the same data. A stream is an unbounded sequence of events (every change recorded). A table is the current state — the latest value per key (materialized view of the stream). Capture a table's change log → you get a stream (CDC). Aggregate a stream → you get a table (materialized view). Kafka Streams and ksqlDB leverage this duality natively.

⚠️Drawback

Materializing large tables from streams requires significant state storage in RocksDB

✅Solution

Use changelog compaction in Kafka to limit state size; set appropriate retention for state stores

26

What is the difference between orchestration and choreography?

Answer

Orchestration uses a central coordinator that directs all services — it knows the full workflow, tells each service what to do, and handles failures. Airflow DAGs are orchestration. Choreography has no central coordinator — services react to events from each other. Each service publishes events and subscribes to others' events. Kafka-based microservices are choreography. Orchestration is easier to debug; choreography is more decoupled.

⚠️Drawback

Orchestration: single point of failure (orchestrator). Choreography: hard to trace flow, emergent behavior

✅Solution

High-availability orchestrator (Airflow HA mode); distributed tracing (Jaeger/OpenTelemetry) for choreography debugging

27

How does shared-nothing MPP architecture work in Redshift?

Answer

Shared-nothing MPP: each node has its own CPU, memory, and storage — no sharing. Leader node receives query, generates execution plan, distributes to compute nodes. Each compute node processes its slice of data in parallel. Nodes send partial results to leader, which aggregates and returns the final result. Distribution key determines which node each row lives on — choosing the right distribution key avoids cross-node shuffles.

⚠️Drawback

Poor distribution key choice causes data skew — one node does all the work; cross-node joins are expensive (data must be broadcast or redistributed)

✅Solution

Choose high-cardinality distribution key matching the most common join column; use EVEN distribution for tables without a clear join key

28

What is tiered storage and how is it used in data architectures?

Answer

Tiered storage matches data access patterns to cost-appropriate storage classes. Hot (S3 Standard, NVMe): last 7-30 days, frequent queries, highest cost. Warm (S3 Standard-IA, Azure Cool): 30-90 days, occasional access. Cold (S3 Glacier): 90+ days, rare access, very low cost. S3 Lifecycle policies automate tier transitions. Can reduce storage cost by 80-95% for data older than 90 days.

⚠️Drawback

Cold storage has retrieval latency (minutes to hours for Glacier) — queries on cold data are very slow

✅Solution

Keep hot/warm tiers in query-optimized format (Delta/Iceberg); use Glacier only for pure compliance archival that rarely needs to be queried

29

Explain the Saga Pattern in distributed systems.

Answer

The Saga Pattern manages long-running distributed transactions across multiple services without a global ACID transaction. Each step is a local transaction. If a step fails, compensating transactions undo previous steps. Two implementations: Choreography Saga (services react to events, no central coordinator) and Orchestration Saga (Saga Orchestrator directs each step and handles compensation). Used for order fulfillment, travel booking, multi-service workflows.

⚠️Drawback

Compensating transactions are difficult to design correctly; partial failures leave system in intermediate states; hard to test all failure scenarios

✅Solution

Implement compensating actions for every step upfront; use idempotent operations; thorough chaos testing of failure scenarios

30

What is vectorized execution and how does it improve performance?

Answer

Vectorized execution processes batches of column values (vectors) using CPU SIMD instructions rather than one row at a time. A single SIMD instruction operates on 256+ bits simultaneously — comparing 8 integers at once instead of 1. Apache Arrow provides the in-memory columnar format enabling vectorized operations. DuckDB, Snowflake, and BigQuery use vectorized engines to achieve 10-100x speedup over row-at-a-time processing.

⚠️Drawback

Vectorized code is more complex to write and debug; benefits vary by query type (aggregations benefit most, row-level updates benefit less)

✅Solution

Use columnar-first tools (DuckDB, Snowflake, ClickHouse) that implement vectorized execution internally — no manual optimization needed

31

How do you handle late-arriving data in stream processing?

Answer

Late data arrives after its event-time window has already closed. Three strategies: (1) Discard — simplest, lose the late event. (2) Allowed Lateness — keep the window open extra time (e.g., 5 minutes) to accept late events and update results. (3) Side Output — route late events to separate stream for manual reprocessing or separate aggregation.

⚠️Drawback

Allowed Lateness increases state size and delays result finalization; long lateness windows negate real-time advantage

✅Solution

Set lateness based on measured P99 event delay from monitoring; use Side Output for anything beyond the allowed window

32

What is backpressure in streaming systems?

Answer

Backpressure is a flow control mechanism where a slow downstream consumer signals upstream producers to reduce their sending rate, preventing buffer overflow and OOM errors. Flink has built-in credit-based backpressure: if a downstream task's buffer is full, upstream tasks automatically reduce their rate. Kafka consumer implements backpressure by controlling how many records to poll (max.poll.records).

⚠️Drawback

Backpressure propagates upstream — if one sink is slow, it can slow the entire pipeline

✅Solution

Monitor consumer lag; auto-scale consumer group with Kubernetes HPA; isolate slow sinks to separate consumer groups

33

What is Master Data Management (MDM) and why is it important?

Answer

MDM creates and maintains a single, authoritative, trusted view (golden record) of key business entities — customers, products, employees, locations. Without MDM, different systems have different versions of the same customer (Salesforce vs CRM vs billing — three different email addresses for Alice). MDM deduplicates, matches, and merges records to produce one golden record distributed to all systems.

⚠️Drawback

Entity resolution (matching records across systems) is hard — especially with dirty data; high implementation cost

✅Solution

Use probabilistic matching for deduplication; start with most critical entity (customer) before expanding; tools: Informatica MDM, Reltio, AWS Entity Resolution

34

Explain the circuit breaker pattern in data pipelines.

Answer

A circuit breaker prevents a pipeline from continuously hammering a failing downstream service. In Closed state: calls pass through, failures counted. After threshold: circuit trips to Open state — calls blocked, fallback used. After reset timeout: moves to Half-Open — test calls allowed. If successful, back to Closed. If failed, re-Open. Prevents cascading failures where a slow DB causes an entire pipeline to hang.

⚠️Drawback

Fallback responses may be stale or incomplete; choosing threshold values requires tuning

✅Solution

Monitor circuit state and alert when open; implement graceful degradation in fallback logic; tune threshold based on service SLA

35

What is the difference between a Data Mart and a Data Warehouse?

Answer

A Data Warehouse is an enterprise-wide repository for all business data — all domains, all history, single source of truth. A Data Mart is a subset of the warehouse scoped to a single department or domain (Sales Mart, Finance Mart). Data Marts improve query performance for domain-specific users by pre-joining and pre-aggregating relevant data. Dependent marts source from the warehouse; independent marts source directly from operational systems (risky — inconsistent definitions).

⚠️Drawback

Data Mart proliferation: 20 independent marts with conflicting metric definitions creates 'which number is correct?' confusion

✅Solution

Enforce dependent marts sourced from a single trusted warehouse; document metric definitions in a data catalog

36

How do you design for GDPR compliance in a data platform?

Answer

GDPR compliance in data platforms requires: (1) Data inventory — catalog all tables containing PII (automated PII detection via Macie/DLP), (2) Data minimization — only collect necessary PII, (3) Encryption at rest and in transit, (4) Row-level and column-level masking for non-privileged roles, (5) Right to erasure — delete or anonymize specific user's data across all tables (row-level delete via Iceberg/Delta), (6) Data retention policies — auto-delete after retention period, (7) Audit logs — track who accessed what data.

⚠️Drawback

Right to erasure in immutable data lake is technically complex; deleting from Kafka topic history is difficult

✅Solution

Use tombstone events in Kafka for logical deletes; Delta/Iceberg row-level delete + VACUUM to physically remove data; encrypt PII with per-user keys — deleting the key makes data unreadable

37

What is the fan-out pattern and when is it used?

Answer

Fan-out distributes a single event or message to multiple independent consumers simultaneously. In Kafka, fan-out is implemented via multiple consumer groups on the same topic — each group reads independently at its own pace. Used when multiple downstream systems need the same event: analytics pipeline, notification service, inventory system, and fraud detection all consuming 'order_placed' independently.

⚠️Drawback

One slow consumer group grows its lag without blocking others but does consume Kafka storage until it catches up

✅Solution

Monitor per-consumer-group lag; set retention policies to limit Kafka storage; use separate topics for slow consumers if needed

38

What is Apache Arrow and why is it important?

Answer

Apache Arrow is an in-memory columnar data format that enables zero-copy reads between systems (Spark, Pandas, DuckDB, Parquet reader) without serialization/deserialization overhead. A Pandas DataFrame in Arrow format can be passed to a Spark job or DuckDB query without copying or converting. Arrow is the foundation for modern vectorized execution engines and enables fast data exchange in the modern data stack.

⚠️Drawback

Arrow format is in-memory — not a storage format. Must convert to/from Parquet for persistence.

✅Solution

Arrow + Parquet are complementary — Arrow for in-memory processing, Parquet for durable storage. Use PyArrow for efficient conversion between both.

39

Explain the difference between hot, warm, and cold storage tiers.

Answer

Hot storage (S3 Standard, Redis, NVMe): high cost, low latency, for frequently accessed current data. Warm storage (S3 Standard-IA, Azure Cool): medium cost, slightly higher retrieval latency, for data accessed occasionally (monthly reports). Cold storage (S3 Glacier, Azure Archive): very low cost, minutes-to-hours retrieval, for rarely accessed archival data. S3 lifecycle policies automate moving data between tiers based on age.

⚠️Drawback

Cold storage retrieval can take hours; querying cold data is impractical for analytics

✅Solution

Only move truly archival data to cold tier; keep compliance data queryable in warm tier (Delta/Iceberg); restore cold data to hot before querying

40

What are the key anti-patterns in data architecture?

Answer

Key anti-patterns: (1) Data Swamp: data lake without governance, catalog, or quality — nobody trusts or finds data. (2) Spaghetti pipelines: point-to-point pipelines between every system — N*(N-1) integrations, brittle. (3) Over-engineering: building Data Mesh microservices for a startup. (4) No idempotency: duplicate data on reruns. (5) Missing schema management: source changes silently break downstream. (6) Ignoring data skew: one Spark task holds up entire stage.

✅Solution

Enforce data catalog from day 1; use central event bus; start with simple architecture and evolve; design for idempotency; implement schema contracts; use AQE for skew handling

41

How does a Schema Registry prevent pipeline failures?

Answer

A Schema Registry stores and versions message schemas (Avro/Protobuf). When a producer registers a new schema, the registry validates it against configured compatibility rules (BACKWARD, FORWARD, FULL). If a breaking change (removing a required field) is attempted, the registry rejects it before it reaches Kafka — preventing consumers from failing on deserializing incompatible messages.

⚠️Drawback

Schema Registry becomes critical infrastructure — if it goes down, producers and consumers cannot serialize/deserialize

✅Solution

Run Schema Registry in HA mode (multiple replicas); cache schemas locally in producers/consumers

42

What is the Modern Data Stack (MDS) and what tools comprise it?

Answer

The Modern Data Stack is a cloud-native, modular data architecture using SaaS tools. Core components: Ingestion (Fivetran/Airbyte), Storage (Snowflake/BigQuery), Transformation (dbt), Orchestration (Airflow/Dagster), BI (Looker/Tableau), Reverse ETL (Census/Hightouch), Observability (Monte Carlo/Soda), Catalog (DataHub). Each component is interchangeable — the stack is composable.

⚠️Drawback

SaaS costs compound — Fivetran + dbt Cloud + Snowflake + Monte Carlo can be very expensive at scale

✅Solution

Use open-source alternatives (Airbyte + dbt Core + self-managed Postgres/DuckDB) for cost efficiency; evaluate build vs buy at each layer

43

How do you design for high availability in a data pipeline?

Answer

High availability design: (1) No single points of failure — Kafka replication factor >= 3, Airflow HA with multiple schedulers, multi-node Spark cluster, (2) Idempotent pipelines — safe to restart on failure, (3) Dead letter queues — failed events don't block pipeline, (4) Circuit breakers — don't hammer failing services, (5) Checkpointing in Flink — restart from checkpoint not beginning, (6) Multi-AZ deployments for all critical infrastructure.

⚠️Drawback

HA infrastructure is 2-3x the cost of single-node; complexity increases with each HA component

✅Solution

Prioritize HA for critical pipelines (financial data, customer-facing); use managed cloud services (MSK, MWAA) which provide HA out of the box

44

What is event sourcing and how does it differ from traditional storage?

Answer

In traditional storage, only the current state is stored (UPDATE customers SET city = 'NYC' — old city is lost). In event sourcing, every state change is stored as an immutable event (customer_moved event with old and new city). Current state is derived by replaying all events. Benefits: full audit trail, time travel, ability to replay and recompute state, event-driven architecture alignment. Kafka is a natural event store.

⚠️Drawback

Replaying all events to get current state is slow for entities with thousands of events

✅Solution

Periodic snapshots of current state — replay only from latest snapshot, not from beginning

45

How do you optimize a slow dbt model?

Answer

Optimization steps: (1) Use incremental materialization — only process new/changed rows, not full table, (2) Add indexes or cluster keys to filter columns, (3) Push filters as early as possible in the model SQL, (4) Use ref() and source() macros correctly to leverage warehouse caching, (5) Avoid SELECT * — select only needed columns, (6) Use warehouse-specific optimizations (Snowflake cluster keys, BigQuery partition/cluster), (7) Profile with EXPLAIN ANALYZE to identify expensive operations.

⚠️Drawback

Incremental models require a unique key and updated_at column; late-arriving data may be missed

✅Solution

Add --full-refresh flag to periodic full reloads; implement is_incremental() checks in model SQL

What is the role of a data catalog in a data platform?

Answer

A data catalog is the search engine and inventory for all data assets. It enables: data discovery (find the right dataset), understanding (business definitions, column descriptions), trust (data quality scores, freshness), governance (PII tags, access policies), and lineage (upstream/downstream dependencies). Without a catalog, data engineers spend hours asking 'where is the customer revenue table?' and 'is this data reliable?'

⚠️Drawback

Catalog becomes stale if not automatically populated — manual documentation doesn't scale

✅Solution

Auto-ingestion from Airflow, Spark, dbt, Kafka via OpenLineage/OpenMetadata integrations; ownership-based incentives for documentation

How would you design a real-time fraud detection pipeline?

Answer

Architecture: Transaction events → Kafka → Flink (stateful stream processing) → fraud scoring. Flink jobs: (1) Join transaction with user profile (enrichment from Redis/Cassandra lookup), (2) Compute features over time windows (5 transactions in 1 min, unusual amount, unusual location), (3) Call ML model inference, (4) If score > threshold → emit fraud alert to Kafka alerts topic → block transaction via API. All sub-second latency.

⚠️Drawback

False positives block legitimate transactions; model drift over time reduces accuracy; state management for user features is complex

✅Solution

A/B test fraud thresholds; shadow mode — run model without blocking, compare to rule-based system; scheduled model retraining pipeline; use Feature Store for consistent features between training and serving

Explain the concept of data freshness and how to monitor it.

Answer

Data freshness measures how up-to-date data is — the lag between when an event occurred and when it's available for querying. Key metrics: source-to-bronze lag, bronze-to-silver lag, pipeline completion time vs SLA. Monitoring: (1) MAX(updated_at) check — alert if no new data after X hours, (2) Row count anomaly — 0 rows today when 50K expected, (3) Pipeline SLA monitoring in Airflow (sla_miss_callback), (4) Observability tools (Monte Carlo, Soda).

⚠️Drawback

Legitimate business events (weekends, holidays) trigger false freshness alerts

✅Solution

Implement business-calendar-aware SLAs; different freshness thresholds for weekdays vs weekends; expected row count ranges per day-of-week

What is the difference between push and pull ingestion patterns?

Answer

Pull ingestion: the pipeline initiates data retrieval — Airflow runs a SELECT query, API polling, SFTP file download. Simple, consumer-controlled, but higher latency (depends on poll frequency) and can overload source. Push ingestion: source system sends data proactively — webhooks, Kafka producers, CDC event streams. Lower latency, looser coupling, but consumer has less control over timing and must handle variable load.

⚠️Drawback

Pull: misses deletes, polling overhead on source. Push: consumer must handle bursts, event loss if consumer is down

✅Solution

Pull: CDC (log-based) to capture deletes. Push: Kafka as durable buffer to handle consumer downtime and bursty load

How do you choose between a Data Warehouse, Data Lake, and Data Lakehouse?

Answer

Data Warehouse (Snowflake/BigQuery): choose when data is structured, team is SQL-heavy, primary use case is BI/reporting, strong governance needed. Data Lake (S3+Hive): choose when storing diverse raw data types (logs, images, unstructured), need cheap archival storage, or primarily for data science exploration. Data Lakehouse (Delta/Iceberg): choose when you need both BI performance AND ML flexibility, want ACID on lake-scale storage, have both batch and streaming workloads, or are building a modern unified platform from scratch.

⚠️Drawback

Lakehouse adds operational complexity vs pure warehouse; requires knowledge of table formats, compaction, vacuum

✅Solution

Use managed lakehouse platforms (Databricks, Snowflake with Iceberg) that abstract operational complexity; start with warehouse-only if team is SQL-focused and evolve to lakehouse as ML needs grow
