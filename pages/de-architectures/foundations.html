What is Data Architecture

Data Architecture defines the models, policies, rules, and standards that govern how data is collected, stored, processed, and accessed across an organization. It is the blueprint that aligns data assets with business goals, ensuring data is reliable, available, secure, and scalable.

📌

Key Components

  • Data Sources — databases, APIs, files, streams, SaaS tools
  • Ingestion Layer — how data enters the system (batch, streaming, CDC)
  • Storage Layer — where data lives (data lake, warehouse, lakehouse)
  • Processing Layer — how data is transformed (Spark, Flink, dbt)
  • Serving Layer — how data is consumed (dashboards, APIs, ML models)
  • Governance Layer — policies, security, lineage, quality
📌

Goals

  • Single source of truth for business data
  • Scalable to handle growing data volumes
  • Reliable with minimal downtime and data loss
  • Secure with proper access control
  • Cost-efficient storage and compute

Evolution of Data Architectures

Data architectures have evolved in response to growing data volumes, real-time requirements, and the shift to cloud.

📌

Timeline

1
1970s–1990s

Relational Databases & Data Warehouses

Centralized RDBMS. ETL pipelines feeding structured warehouses. IBM DB2, Oracle, Teradata.

⚠

Cannot handle unstructured data, poor scalability, expensive

2
2000s

Data Warehouses + ETL Tools

Enterprise data warehouses with ETL tools (Informatica, SSIS). Star/snowflake schemas.

⚠

Rigid schemas, slow to change, high cost

3
2006–2012

Hadoop & Big Data

HDFS + MapReduce for distributed storage/processing. Handle petabytes of unstructured data.

⚠

Complex operations, slow MapReduce, no ACID, hard to use

4
2011–2015

Lambda Architecture

Batch layer (Hadoop) + Speed layer (Storm/Spark Streaming) + Serving layer.

⚠

Two codebases to maintain (batch and stream logic duplicated)

5
2014–2018

Kappa Architecture & Data Lakes

Single stream-first pipeline. Data lakes (S3/ADLS) as cheap central storage.

⚠

Data swamps, no ACID, poor query performance, schema chaos

6
2018–Present

Data Lakehouse & Modern Data Stack

Delta Lake, Iceberg, Hudi bring ACID to lakes. Cloud warehouses (Snowflake, BigQuery). dbt for transformation.

⚠

Complexity, vendor lock-in concerns

7
2020–Present

Data Mesh & Data Fabric

Decentralized domain-owned data products. Federated governance. Real-time everywhere.

⚠

Organizational change required, not a purely technical solution

CAP Theorem

CAP Theorem (Brewer's Theorem) states that a distributed system can guarantee at most TWO of three properties simultaneously: Consistency, Availability, and Partition Tolerance.

📋

Properties

1

Consistency (C)

Every read receives the most recent write or an error. All nodes see the same data at the same time.

2

Availability (A)

Every request receives a response (not an error), though it may not be the most recent data.

3

Partition Tolerance (P)

The system continues operating even if network partitions (communication failures between nodes) occur.

🔀

Cap Combinations

1

CP (Consistency + Partition Tolerance)

Sacrifices availability. On partition, system returns error rather than stale data.

Use Case

Financial systems where accuracy is critical

HBaseZookeeperMongoDB (with strong consistency)
2

AP (Availability + Partition Tolerance)

Sacrifices consistency. On partition, returns possibly stale data but stays available.

Use Case

Shopping carts, user sessions, high-availability apps

CassandraDynamoDBCouchDB
3

CA (Consistency + Availability)

Only possible without network partitions — not realistic in distributed systems.

Use Case

Single-node systems where partition tolerance is not needed

Single-node RDBMS (PostgreSQL, MySQL)
📌

Pacelc Extension

NamePACELC
DescriptionExtends CAP: even without partitions (E), a system must trade off Latency (L) vs Consistency (C).
NoteMore practical than CAP for modern distributed system design
📌

De Relevance

When designing data pipelines, CAP determines: Should the pipeline prioritize delivering data immediately (AP) or only deliver when fully consistent (CP)? Kafka is AP for message delivery. PostgreSQL is CA for single-node. Cassandra is AP for high-write workloads.

Data Flow Patterns

Data flow patterns describe how data moves between systems in an architecture.

📌

Patterns

1

Point-to-Point

Direct connection between source and destination system. Simple but creates tight coupling and spaghetti integrations at scale.

Diagram

Source A → Destination B

Drawback

N*(N-1) connections for N systems. Brittle when sources or destinations change.

2

Hub and Spoke

Central hub (data warehouse or message broker) that all systems connect to. Reduces point-to-point connections.

Diagram

Sources → Central Hub → Destinations

Drawback

Hub becomes a single point of failure and bottleneck.

3

Pipeline (Sequential)

Data flows through a sequence of processing stages. Output of each stage is input to the next.

Diagram

Ingest → Validate → Transform → Enrich → Load → Serve

Drawback

Failure in any stage blocks all downstream stages.

4

Event-Driven / Pub-Sub

Producers publish events to a topic. Multiple consumers subscribe independently. Decoupled.

Diagram

Producer → Kafka Topic → [Consumer A, Consumer B, Consumer C]

Drawback

Harder to debug, eventual consistency, ordering guarantees complex.

5

Federated

Data remains in domain-owned systems. A query engine federates across multiple sources at query time.

Diagram

Query Engine → [Domain DB A, Domain DB B, Domain DB C]

Drawback

High query latency, complex governance across domains.
