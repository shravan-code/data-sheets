Storage Patterns

6 topics

Partitioning Strategies

Partitioning physically divides data into separate folders/files based on column values. Queries that filter on partition columns skip irrelevant partitions (partition pruning), dramatically improving performance.

📌

Types

1

Date/Time Partitioning

Most common. Partition by year/month/day.

Example

s3://lake/orders/year=2024/month=01/day=15/

Best For

Time-series data, event logs, transactional data

2

List Partitioning

Partition by categorical values.

Example

s3://lake/orders/region=us-east/

Best For

Geographic data, department/category splits

3

Range Partitioning

Partition by numeric ranges.

Example

s3://lake/customers/age_bucket=18-25/

Best For

Numeric ranges, score bands

4

Hash Partitioning

Distribute rows evenly via hash function.

Example

Spark: df.repartition(200, col('customer_id'))

Best For

Even distribution for joins, avoiding data skew

📌

Partition Pruning Example

QuerySELECT * FROM orders WHERE order_date = '2024-01-15'
Without PartitioningScans 100M rows (full table)
With Date PartitioningScans only 274K rows (one day's partition)
Speedup~365x faster
⚠️

Drawbacks

  • •Over-partitioning creates too many small files (small file problem)
  • •Wrong partition key = no pruning benefit
  • •Partition column cardinality must be reasonable (not unique user IDs)
✅

Solutions

  • ✓Partition on low-to-medium cardinality columns
  • ✓Combine with Z-ordering (Databricks) for high-cardinality filter columns
  • ✓Run compaction regularly to merge small partition files

Bucketing

Bucketing distributes data into a fixed number of buckets using a hash of the bucket column. Unlike partitioning, buckets are fixed-count files. Improves join performance when both tables are bucketed on the join key.

💡

Example

Hive SqlCREATE TABLE orders_bucketed CLUSTERED BY (customer_id) INTO 64 BUCKETS STORED AS PARQUET;
BenefitJOIN between orders_bucketed and customers_bucketed (both bucketed on customer_id): no shuffle needed — matching buckets are joined directly
SpeedupEliminates shuffle in distributed joins — major performance gain
⚠️

Drawbacks

  • •Fixed bucket count — choosing wrong count causes skew or overhead
  • •Both tables must be bucketed on same key and count for join optimization
  • •Not supported in all query engines

Compaction

Compaction merges many small files into fewer large files. The small file problem occurs when streaming or frequent micro-batch writes create thousands of tiny files, degrading query performance.

📌

Small File Problem

CauseSpark Structured Streaming writes one small file per micro-batch. 1000 micro-batches = 1000 files per partition.
ImpactEach file = one HDFS/S3 metadata call. 1M files = 1M metadata calls = very slow queries.
ExampleBefore compaction: 10,000 files × 1MB = 10GB. After: 10 files × 1GB = 10GB (same data, much faster to query)
📌

Compaction Approaches

1

Delta Lake OPTIMIZE

Merges small files and reorders data by Z-order for better data skipping

Sql

OPTIMIZE delta.`s3://lake/silver/orders` ZORDER BY (customer_id);

2

Iceberg Rewrite Data Files

Merges small files in Iceberg tables

Sql

CALL catalog.system.rewrite_data_files('db.orders');

3

Hudi Inline Compaction

Hudi compacts MOR (Merge-On-Read) delta logs into base files automatically

⚠️

Drawbacks

  • •Compaction is compute-intensive — schedule during low-traffic windows
  • •Compaction rewrites files — metadata versioning grows (need VACUUM too)

Tiered Storage

Different storage tiers offer different cost/performance trade-offs. Data moves between tiers based on access frequency and age.

📌

Tiers

1

Hot Storage

Frequently accessed, low-latency, high-cost

Use Case

Last 7 days of transactional data, real-time dashboards

NVMe SSDRedisS3 StandardSnowflake active warehouse
2

Warm Storage

Occasionally accessed, medium cost

Use Case

Last 90 days, monthly reports

S3 Standard-IAAzure Cool BlobNearline GCS
3

Cold Storage

Rarely accessed, very low cost, higher retrieval latency

Use Case

Compliance archives, 2+ year historical data

S3 GlacierAzure ArchiveGCS Coldline
📌

Lifecycle Policy Example

S3 Lifecycle["Day 0-30: S3 Standard (hot)","Day 31-90: S3 Standard-IA (warm)","Day 91-365…
Cost SavingMoving 1TB from Standard to Glacier Deep Archive: ~$23/month → $1/month

File Formats (Parquet, ORC, Avro, Delta, Iceberg)

Choosing the right file format has major impact on storage cost, query performance, and compatibility.

📌

Formats

1

Columnar

Most widely used analytical format. Columnar storage enables reading only needed columns. Excellent compression.

Format

Parquet

Best For

Analytical queries, data lakes, Spark, Hive, Athena

Compression

Snappy (default), GZIP, ZSTD

Drawback

Not ideal for row-level updates — append-only

2

Columnar

Columnar format optimized for Hive workloads. Better compression than Parquet in some cases. Built-in indexes.

Format

ORC (Optimized Row Columnar)

Best For

Hive-heavy ecosystems

Drawback

Less ecosystem support than Parquet outside Hive

3

Row-based

Row-based format with rich schema (JSON-defined). Excellent for streaming (Kafka) and row-level serialization.

Format

Avro

Best For

Kafka message serialization, schema evolution, write-heavy workloads

Drawback

Poor analytical query performance (must read all columns)

4

Table Format (on Parquet)

Parquet files + transaction log (_delta_log). Adds ACID, time travel, schema enforcement, and DML to data lakes.

Format

Delta Lake

Best For

Databricks-centric lakehouses, ACID on data lake

Drawback

Delta format is Databricks-native (though open-sourced)

5

Table Format (on Parquet/ORC/Avro)

Open table format with rich metadata layer. Supports hidden partitioning, time travel, schema evolution, and row-level deletes.

Format

Apache Iceberg

Best For

Multi-engine environments (Spark + Trino + Flink on same table)

Drawback

More complex catalog management

Compression Strategies

Compression reduces storage size and I/O costs. The right codec balances compression ratio vs CPU overhead.

📌

Codecs

1

Snappy

Ratio

Medium (2-3x)

Speed

Very fast

Best For

Default for Parquet — good balance

2

GZIP

Ratio

High (4-5x)

Speed

Slower

Best For

Archival, cold storage where read speed matters less

3

ZSTD

Ratio

High (4-5x)

Speed

Fast

Best For

Best balance of speed and ratio — modern default

4

LZ4

Ratio

Low (1.5-2x)

Speed

Fastest

Best For

Real-time streaming, speed critical

5

Brotli

Ratio

Very high

Speed

Slowest

Best For

Web static files, maximum compression

📌

Recommendation

Use ZSTD for new Parquet data lake tables — best compression ratio with good read performance. Use Snappy for existing Spark pipelines (default). Use GZIP for cold archival.
