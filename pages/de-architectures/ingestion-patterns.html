Ingestion Patterns

6 topics

Batch Ingestion

Data is collected and loaded in bulk at scheduled intervals (hourly, daily, weekly). The entire dataset or a large chunk is processed at once.

⚡

Characteristics

  • Scheduled (cron, Airflow DAG)
  • High throughput — processes large volumes efficiently
  • Higher latency — data is stale between runs
  • Simple to implement and debug
  • Good for large historical loads and reporting
💡

Example

ScenarioDaily sales ingestion from PostgreSQL to S3
Airflow Dag{"schedule":"0 2 * * *","tasks":["extract_orders: SELECT * FROM orders WHERE upd…
Volume500,000 rows per day, ~200MB Parquet

Tools

Apache AirflowAWS GluedbtTalendFivetran
⚠️

Drawbacks

  • •Data is stale between batch runs
  • •Large batch runs can overload source systems
  • •Failed batches require reprocessing the full batch
✅

Solutions

  • ✓Incremental loads instead of full loads
  • ✓Watermark-based extraction to minimize source load
  • ✓Idempotent writes to safely rerun failed batches

Micro-Batch Ingestion

Data is processed in small batches at very short intervals (every few seconds to minutes). Bridges the gap between batch and true streaming. Used by Spark Structured Streaming's trigger interval.

💡

Example

ScenarioNear-real-time order processing
Spark Codedf = spark.readStream.format('kafka').option('kafka.bootstrap.servers', 'broker:9092').option('subscribe', 'orders').load() df.writeStream.format('delta').option('checkpointLocation', '/chk/orders').trigger(processingTime='30 seconds').outputMode('append').start('s3://lake/silver/orders')
BehaviorEvery 30 seconds: reads all Kafka messages since last trigger, processes, writes to Delta
⚠️

Drawbacks

  • •Not truly real-time — 30-second lag may be unacceptable for some use cases
  • •Checkpoint management adds overhead
  • •Small file problem if trigger interval is too short
✅

Solutions

  • ✓Tune trigger interval based on latency requirements
  • ✓Use Delta Lake compaction to merge small files
  • ✓Use continuous processing mode for sub-second latency

Real-Time / Streaming Ingestion

Events are ingested and processed individually or in very small groups as they arrive, with millisecond-to-second latency.

💡

Flow["User clicks on webpage","Browser sends event to tracking API","API publishes t…

Tools

Apache KafkaApache FlinkAWS KinesisGoogle Pub/SubApache Pulsar
⚠️

Drawbacks

  • •Complex state management for aggregations over time windows
  • •Handling late-arriving or out-of-order data
  • •Higher infrastructure cost (always-on consumers)
  • •Exactly-once delivery is complex to implement correctly
✅

Solutions

  • ✓Watermarks to handle late data
  • ✓Kafka transactions + Flink checkpointing for exactly-once
  • ✓Separate streaming path for real-time and batch for historical

Change Data Capture (CDC)

CDC captures row-level changes (INSERT, UPDATE, DELETE) from a source database's transaction log and streams them downstream. Enables near-real-time sync without polling queries.

📌

Cdc Types

1

Log-Based CDC

Reads the database transaction/WAL log (PostgreSQL WAL, MySQL binlog). Most efficient. Zero impact on source.

DebeziumAWS DMSFivetran
2

Timestamp-Based CDC

Polls source table for rows where updated_at > last_run_time. Simple but misses hard deletes.

Sql

SELECT * FROM orders WHERE updated_at > :last_watermark

Drawback

Misses DELETE operations. Requires updated_at column.

3

Trigger-Based CDC

Database triggers write changes to an audit table. Captured from audit table.

Drawback

Adds write overhead to source system.

📌

Full Cdc Pipeline

FlowPostgreSQL WAL → Debezium → Kafka (topic: db.orders) → Kafka Connect S3 Sink → Delta Lake → dbt → Gold tables
Latency< 1 second end-to-end
⚠️

Drawbacks

  • •Log-based CDC requires DB-level permissions (REPLICATION role)
  • •Schema changes in source can break CDC pipeline
  • •Initial snapshot for large tables takes significant time
  • •Transaction log retention must be sufficient
✅

Solutions

  • ✓Schema Registry to handle schema evolution in CDC events
  • ✓Monitor log retention and alert before logs are purged
  • ✓Use Debezium's snapshot mode for initial full load

Full Load vs Incremental Load

Two fundamental strategies for loading data from source to destination.

📊

Comparison Table

Featurefull loadincremental
What is loadedEntire dataset every runOnly new or changed rows since last run
Source querySELECT * FROM ordersSELECT * FROM orders WHERE updated_at > :last_watermark
Load timeLong (grows with table size)Short (only delta)
Source system impactHigh (full table scan every run)Low (only reads new rows)
Handles deletesYes (missing rows = deleted)Only with soft deletes or CDC
IdempotencyEasy (truncate + reload)Requires upsert logic
Use whenSmall tables, initial load, no updated_at columnLarge tables, frequent changes, has updated_at or CDC
📌

Incremental Example

Watermark TableCREATE TABLE _etl_watermarks (table_name TEXT, last_loaded_at TIMESTAMP);
Extract SqlSELECT * FROM orders WHERE updated_at > (SELECT last_loaded_at FROM _etl_watermarks WHERE table_name = 'orders')
Update WatermarkUPDATE _etl_watermarks SET last_loaded_at = NOW() WHERE table_name = 'orders'
