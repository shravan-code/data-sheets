System Design for Data

Conversational breakdowns of complex data architectures. What they are, how they work, and when to use them.

1. Core Concepts

Data Ingestion

Well, before you can analyze data, you have to actually acquire it. Data ingestion is the process of moving data from where it was created (source) to a place where it can be stored and analyzed (destination).

a. What Is Batch Processing???

Batch processing is a strategy that says: Let's gather a huge chunk of data over time, and process it all at once on a schedule (like every midnight).

b. What Is Stream Processing???

Stream processing acts immediately. As soon as a single event occurs (like a user clicking a button), it grabs that data and processes it right away.

Think of it like getting your mail. Batch is like checking your physical mailbox once a day and handling all 10 envelopes at once. Streaming is like getting an SMS text notification and reading it the precise second your phone buzzes.

When to Use Batch vs Streaming

  • Use Batch: For huge analytical jobs that take hours (like calculating monthly payroll or training an ML model).
  • Use Streaming: For things that require millisecond decisions (like detecting credit card fraud or updating a live sports scoreboard).
Data Source │ ├─ [Wait 24 Hours] → [Process 1,000,000 Rows] → DWH (Batch) │ └─ [Arrives Now] → [Process 1 Row] → Dashboard (Stream)

CAP Theorem & Consistency Models

In distributed systems (systems running across many computers), networks will eventually fail. The CAP Theorem forces you to pick your poison when that happens.

a. What Is the CAP Theorem???

It states that when a network partition (P) occurs—meaning servers lose connection to each other—you can only guarantee either Consistency (C) or Availability (A). You cannot have both.

b. Strong vs Eventual Consistency???

Strong consistency ensures that once a database is updated, every subsequent read gets the new data immediately (C over A). Eventual consistency says: "We'll reply fast with whatever old data we have right now to stay alive, and we'll sync up the correct data across servers in the background eventually" (A over C).

Think of Strong Consistency like a shared bank account. If you withdraw $50, your partner's debit card must immediately be declined from withdrawing another $50. No mistakes allowed. Think of Eventual Consistency like YouTube view counts. If you like a video, it might say 1,000 views for you, but 998 views for someone else. It's not perfectly synced, but it constantly stays online, and syncs eventually.
Network breaks between Node 1 and Node 2! [Node 1 gets request to update User Profile] Option A (Strong Consistency): "I can't talk to Node 2! Rejecting the update to prevent mismatch!" (Downtime) Option B (Eventual Consistency): "I'll accept the update and stay online! I'll tell Node 2 later when the network fixes!" (Data Mismatch)

2. Data Storage & Modeling

Data Lake vs Data Warehouse

You have massive amounts of data. Where do you put it? The eternal battle of storage usually fundamentally comes down to Lakes vs Warehouses.

a. What Is a Data Lake???

A Data Lake is a massive, incredibly cheap raw storage bin (like AWS S3). It stores data exactly as it arrives—messy JSONs, CSVs, images, or audio files. There are no rules.

b. What Is a Data Warehouse???

A Data Warehouse is a strictly governed, highly structured relational database (like Snowflake). The data here is pristine, organized into perfect tables, and optimized for business analysts to query using SQL.

Think of a Data Lake like a giant messy warehouse box. You just throw all your receipts, tax forms, and random papers in there quickly so you don't lose them. Think of a Data Warehouse like a highly organized filing cabinet. You physically cannot put a paper inside it unless you meticulously sort it into a specific tabbed folder first.

When to Use Lake vs Warehouse

Nowadays, you usually use both! You dump raw data into the Lake. Then, data engineers clean it, process it, and load only the structured results into the Warehouse for the BI team to use.

Raw API Traffic ──> [Data Lake] (Cheap, Messy, Schema-on-Read) │ (Data Engineer Cleans) ↓ [Data Warehouse] (Expensive, Clean, Schema-on-Write) │ (BI Analyst)

OLTP vs OLAP

Not all databases are created equal. The way you store data drastically changes depending on whether you are writing single transactions, or reading massive analytics.

a. What Is OLTP???

Online Transaction Processing. These systems (like Postgres or MySQL) are designed for incredibly fast, single-row inserts and updates. They are the backbone of live web applications.

b. What Is OLAP???

Online Analytical Processing. These systems (like BigQuery) are designed to read millions of rows of historical data to perform massive calculations, aggregations, and business intelligence reporting.

Think of OLTP like a cashier checking out a customer at a grocery store. It's one unique transaction happening fast. Think of OLAP like the store manager at the end of the year asking: "What was the total revenue of all apples sold across 50 stores in August?"

File Formats (Row vs Columnar)

In Data Engineering, saving a file as `.csv` or `.txt` simply doesn't cut it when you are dealing with Petabytes of data. Advanced file formats exist to compress data and make querying lightning fast.

a. What Are Row-Based Formats (CSV / JSON / Avro)???

These formats store data row-by-row. If you have 100 users, it saves User 1's Name, Age, and Email, then moves to User 2. Avro is the king of row-based formats in big data because it enforces a strict schema and is extremely fast for writing heavy data streams (like Kafka).

b. What Are Columnar Formats (Parquet / ORC)???

Columnar formats physically store all identical columns together on the hard drive. It saves all 100 Names, then all 100 Ages, then all 100 Emails. Parquet is the gold standard for analytics (OLAP). Why? Because if a BI analyst runs SELECT SUM(revenue), Parquet only reads the 'revenue' block from the hard drive and completely ignores the other columns, saving massive I/O time. Grouping identical data types together also allows for insane compression rates.

Think of Row-Based storage like reading a book left to right, sentence by sentence (great if you need the full story). Think of Columnar storage like reading only the first word of every page to find a specific noun (great if you are just counting words and don't care about the story).
Data Table: | ID | Age | Name | |----|------|------| | 1 | 25 | Bob | | 2 | 30 | Amy | Row-Based Disk Layout (Great for writing/inserting 1 full user): [1, 25, Bob] [2, 30, Amy] Columnar Disk Layout (Great for reading just the "Age" column): [1, 2] [25, 30] [Bob, Amy]

3. Data Processing

Lambda & Kappa Architecture

When building systems that need to handle both historical accuracy and real-time dashboards, architects debate between two massive structural patterns.

a. What Is Lambda Architecture???

Lambda maintains two strictly parallel pipelines. 1) A slow Batch Layer that calculates everything perfectly. 2) A fast Speed (Stream) layer that calculates real-time approximations. The final serving layer queries both and merges the answers.

b. What Is Kappa Architecture???

Kappa says: Maintaining two different codebases for batch and stream is annoying. Let's just treat everything as a stream. A batch job is simply "streaming" a huge chunk of historical events from an immutable log.

Think of Lambda like watching a live sports game on TV (fast but low resolution approximation), while simultaneously recording the 4K Blue-Ray onto disk to watch perfectly tomorrow. Think of Kappa as just upgrading your internet so the live TV effectively is the 4K stream entirely.
Lambda Architecture: ┌──> (Batch Layer: Slow, Accurate) ──┐ Incoming Data ─┤ ├─> Serving Layer └──> (Speed Layer: Fast, Approx) ────┘ Kappa Architecture: Incoming Data ────> (Stream Layer) ───────────────> Serving Layer

4. Scalability & Performance

Partitioning & Sharding

When a database table gets too large (billions of rows), queries become painfully slow. You have to physically break the data apart so the computer doesn't have to scan the whole thing.

a. What Is Partitioning???

Partitioning splits a massive logical table into smaller physical pieces on the same machine (e.g. splitting a "Sales" table into twelve mini tables, one for each "Month").

b. What Is Sharding???

Sharding takes partitioning and distributes it across multiple different servers. Server 1 holds Users A-M. Server 2 holds Users N-Z.

Think of Partitioning like taking a massive 1,000-page dictionary and adding "A-to-Z" chapter tabs so you can jump to a page fast. Think of Sharding like literally ripping the dictionary in half and giving the A-M half to your friend to hold across the room.
Big Table [A, B, C, D] Partitioning (Same Machine): Folder: 2025-Jan [A, B] Folder: 2025-Feb [C, D] Sharding (Different Machines): Server_US : DB [A, B] Server_EU : DB [C, D]

5. Messaging & Integration

Pub/Sub & Message Queues

As systems grow, having APIs talk directly to each other causes failures. If Server A directly calls Server B, and Server B is offline... Server A crashes too. We use brokers to decouple them.

a. What Is a Message Queue (e.g. SQS)???

A queue is a 1-to-1 shock absorber. A producer drops a task in the queue. A single consumer picks it up, successfully processes it, and deletes it from the queue.

b. What Is Pub/Sub (e.g. Kafka)???

Publish-Subscribe is a 1-to-Many broadcast. A producer drops an event into a "Topic" (like an FM radio station). Countless independent consumers can "tune in" and read that same event without affecting each other.

Think of a Queue like placing an order at McDonald's. You put the ticket in the queue, exactly one chef takes the ticket, cooks the burger, and the ticket is thrown away. Think of Pub/Sub like posting a tweet. You publish it once, and a million completely independent followers can read it.
Message Queue (SQS): App ──> [Queue] ──> Worker 1 (Message deleted) Pub/Sub (Kafka): App ──> [Topic] ──┬──> Analytics Service (Reads it) ├──> Fraud Service (Reads it) └──> Email Service (Reads it)

6. Reliability & Fault Tolerance

Retry & Dead Letter Queues (DLQ)

Well, in data engineering, failures are inevitable. Network timeouts, malformed records, rate limits — something will go wrong eventually. What matters is how you handle it.

a. What Is Retry Logic???

Retry logic is a strategy that says: If this task fails, let's try again before giving up (often using exponential backoff to not overwhelm the system).

b. What Is a Dead-Letter Queue (DLQ)???

A Dead-Letter Queue is a special queue where you send events or messages that repeatedly fail to process — so they don’t block the pipeline. Think of it like a “quarantine bin” for bad data.

Incoming Event Stream ↓ Validate Schema ↓ ┌─────────────┐ │ Enrich API │ ← Retry on timeout └────┬────────┘ ↓ Bad Event? → Yes → Send to DLQ ↓ Save to DWH

Idempotency

Because retries are guaranteed, systems will occasionally send the exact same data to your database twice. Your system must be protected against this.

a. What Is Idempotency???

Idempotency ensures that running the exact same operation multiple times yields the exact same final result, without applying unintended side effects (like duplicating rows).

Think of standard insertion like pushing an "Add $1" button. If you push it twice, you get $2 (Not Idempotent). Think of Idempotency like pushing an elevator button. If you push floor 5, and then push it again just to be safe, the elevator still just goes to floor 5 once.

7. Schema Evolution

Schema Evolution

Software developers constantly change applications. Today the sign-up form has 3 fields, tomorrow it has 4. If your downstream data pipeline expects exactly 3 columns, it will crash violently tomorrow.

a. What Is Schema Evolution???

It's the architectural implementation (usually through formats like Avro paired with a Schema Registry) that dictates how a pipeline gracefully handles missing columns, new columns, or changed data types without failing the entire batch.

Day 1: { "user", "age" } → Pipeline succeeds. Day 2: { "user", "age", "email" } → Pipeline succeeds (Schema Evolution adds column). Day 3: { "user" } → Pipeline succeeds (Schema Evolution sets 'age' backfill to Null).

8. Orchestration (DAGs)

Workflow Scheduling & DAGs

When you have 500 different scripts that clean data, train models, and update dashboards, you can't just run them randomly. You need a conductor. Enter tools like Apache Airflow.

a. What Is a DAG???

A Directed Acyclic Graph represents the execution flow. "Directed" means arrows point one way. "Acyclic" means no infinite loops. It strictly outlines dependencies, ensuring that the "Generate Dashboard" script refuses to run until the "Clean Daily Data" script successfully finishes.

[Extract DB Users] ─────┐ ↓ [Extract API Logs] ──> [Join Data] ──> [Update Dashboard]

9. Caching

Caching Strategies

Databases are slow because they read from hard drives. Caching solves this by temporarily storing wildly popular data directly in memory (RAM) so it can be retrieved instantly. The catch? Memory is expensive and volatile.

a. What Is Cache-Aside???

The application explicitly manages the cache. It asks the Cache for data. If it's missing (a "miss"), the app asks the Database, hands the data to the user, and retroactively updates the Cache.

b. What Is Write-Through???

The application writes data strictly to the Cache. The Cache then synchronously writes that data to the Database before confirming success. It guarantees data is always up-to-date, but writes are slower.

Think of an Encyclopedia as a Database, and your brain as a Cache. If asked a question, you check your brain first (Cache-Aside). If you don't know it, you go look it up in the Encyclopedia, tell your friend the answer, and memorize it for next time.
Cache-Aside Flow: 1. App reads from [Cache] → Miss! 2. App reads from [Database] → Hit! 3. App writes answer into [Cache] 4. App returns answer to User

10. Security & Governance

IAM & Encryption

Data is the most valuable asset a modern company owns. If you don't lock it down, someone will eventually walk away with it. You need authorization gates and cryptographic locks.

a. What Is IAM (Identity & Access Management)???

IAM is the bouncer. It strictly dictates which users (or automated scripts) are allowed to read, write, or delete specific tables inside your warehouse based on the principle of least privilege.

b. Encryption At Rest vs In Transit???

In Transit: Encrypts the "pipe" (like HTTPS/TLS). If a hacker intercepts the Wi-Fi signal while data moves from AWS to your laptop, they just see gibberish. At Rest: Encrypts the hard drive itself. If someone literally breaks into the Google data center and steals the physical SSD, they can't read what's on it.

Think of IAM like checking IDs at the front door of a bank. Think of Encryption In Transit like using an armored car to drive cash between banks. Think of Encryption At Rest like putting the cash inside an indestructible titanium vault once inside the bank.

Data Lineage

When the CEO looks at a dashboard showing revenue dropped by 50%, they will scream at the data team. You must instantly prove where that number came from to debug it.

a. What Is Data Lineage???

Lineage is the chronological map of how a single piece of data mutated over time. It traces a metric on a dashboard all the way backward through 15 intermediate SQL staging tables, straight to the raw PostgreSQL database it originated from.

Tracing lineage of "Total_Rev": [BI Dashboard] ↑ [fact_monthly_revenue] (Aggregated table) ↑ [stg_stripe_payments] (Cleaned JSON) ↑ [raw_api_stripe] (Messy raw ingest)

11. Monitoring

Logs, Metrics & Alerting

If a pipeline breaks in a forest and nobody sets up Datadog, does it make a sound? Yes, it makes the sound of customers canceling their subscriptions. You need to know when things break before the customers do.

a. What Exactly Are Logs???

Logs are immutable, descriptive strings generated when specific events happen or errors crash the code. They answer: "What precisely went wrong here?"

b. What Are Metrics???

Metrics are aggregated, numerical measurements over time. They answer: "Is the system generally healthy compared to yesterday?"

Think of Metrics like looking at the dashboard of your car to see the engine temperature is 30% hotter than normal. It tells you there is a problem. Think of Logs like plugging a diagnostic computer into the car to read out the exact error code: "Faulty Radiator Fan Sensor on Line 42".

c. When to Use Alerting

You hook your metrics up to an Alerting tool (like PagerDuty). If the "Failed API Requests" metric goes above 5% per minute, the system automatically triggers a terrifying phone call to the on-call engineer at 3 AM.

[App] generates Logs & Metrics ↓ [Grafana / Datadog] displays a massive spike in memory usage ↓ [Threshold Exceeded > 90%] ↓ [PagerDuty] calls Engineer's cell phone
