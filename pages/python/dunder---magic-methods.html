Section 5: Foundations

Dunder / Magic Methods

Object Representation: __str__ & __repr__

__str__ is for end-users (readable); __repr__ is for developers (detailed/ambiguous). Critical for debugging and logging.

class User:
    def __init__(self, name): self.name = name
    def __repr__(self): return f'User(name={self.name!r})'
    def __str__(self): return f'User: {self.name}'

u = User('Alice')
print(repr(u))
print(str(u))
Output
User(name='Alice')
User: Alice

Arithmetic Dunders: __add__, __mul__

Implement mathematical operations for your objects. Allows using + and * operators directly on custom instances.

class Point:
    def __init__(self, x, y): self.x, self.y = x, y
    def __add__(self, other): return Point(self.x + other.x, self.y + other.y)
    def __repr__(self): return f'({self.x}, {self.y})'

p1 = Point(1, 2); p2 = Point(3, 4)
print(p1 + p2)
Output
(4, 6)

Comparison Dunders: __eq__, __lt__

Custom objects don't have built-in comparison logic. Implement these to enable sorting and equality checks.

class Task:
    def __init__(self, p): self.priority = p
    def __lt__(self, other): return self.priority < other.priority

tasks = [Task(3), Task(1)]
print(sorted(tasks)[0].priority)
Output
1

Container Dunders: __len__, __getitem__

Make your objects behave like lists or dictionaries. Allows using len() and square-bracket indexing [].

class Deck:
    def __init__(self): self.cards = ['A', 'K', 'Q']
    def __len__(self): return len(self.cards)
    def __getitem__(self, i): return self.cards[i]

d = Deck()
print(len(d), d[0])
Output
3 A

Callable Objects: __call__

Allows an instance to be called like a function. Useful for creating stateful functions or decorators.

class Adder:
    def __init__(self, n): self.n = n
    def __call__(self, x): return x + self.n

add5 = Adder(5)
print(add5(10))
Output
15

Interview Mastery

Q: What is the difference between __str__ and __repr__?

A: __str__ is for human-readable output (used by print/str()). __repr__ is for unambiguous developer representation — should ideally return code that recreates the object. If __str__ is not defined, Python falls back to __repr__. Always define __repr__ as minimum.

Q: When does Python call __eq__ vs __hash__?

A: If you override __eq__, Python sets __hash__ to None (making the object unhashable) unless you also define __hash__. Objects that compare equal must have the same hash. The contract: `a == b` implies `hash(a) == hash(b)`.
