Typing Utility Functions

cast, overload, get_type_hints, TYPE_CHECKING

Runtime typing utilities and static analysis helpers.

from typing import cast, overload, get_type_hints, TYPE_CHECKING

# cast - tell type checker the type (no runtime effect)
data = [1, 2, 3]
data_as_list = cast(list[int], data)
print(type(data_as_list))  # still list, cast is a no-op at runtime

# overload - multiple type signatures
@overload
def process(x: int) -> int: ...
@overload
def process(x: str) -> str: ...
def process(x):
    if isinstance(x, int):
        return x * 2
    return x.upper()

print(process(5))
print(process('hello'))

# TYPE_CHECKING - avoid circular imports
if TYPE_CHECKING:
    from collections import OrderedDict  # only imported for type checkers

# get_type_hints - resolve annotations at runtime
def example(x: int, y: str) -> bool:
    pass

print(get_type_hints(example))
Output
<class 'list'>
10
HELLO
{'x': <class 'int'>, 'y': <class 'str'>, 'return': <class 'bool'>}

Interview Mastery

Q:

What is the difference between Protocol and ABC?

A:

ABC (Abstract Base Class) requires explicit inheritance: `class MyClass(MyABC)`. Protocol uses structural subtyping — if an object has the right methods/attributes, it satisfies the Protocol without inheritance. Protocol is the Pythonic way to formalize duck typing. ABCs are better for enforcing that subclasses implement required methods.

Q:

Does Python enforce type hints at runtime?

A:

No. Type hints are purely for static analysis tools (mypy, pyright) and IDE support. Python ignores them at runtime unless you use a library like pydantic or beartype. You can access annotations via __annotations__ or get_type_hints().
