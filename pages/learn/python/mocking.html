Mocking

unittest.mock — Mock, MagicMock, patch

Mocking replaces real objects/functions with controllable fakes for isolated testing.

from unittest.mock import Mock, MagicMock, patch, call
import pytest

# Basic Mock
m = Mock()
m.method(1, 2, key='value')
print(m.method.called)              # True
print(m.method.call_count)          # 1
print(m.method.call_args)           # call(1, 2, key='value')

# Configure return value and side_effect
m2 = Mock()
m2.fetch.return_value = {'data': 42}
print(m2.fetch('url'))              # {'data': 42}

m3 = Mock()
m3.connect.side_effect = ConnectionError('Failed')
try:
    m3.connect()
except ConnectionError as e:
    print(e)

# MagicMock - supports magic methods
mm = MagicMock()
mm.__len__.return_value = 5
print(len(mm))  # 5

# patch - replace real objects in tests
class UserService:
    def get_user(self, user_id):
        import requests
        response = requests.get(f'https://api.example.com/users/{user_id}')
        return response.json()

# Test without real HTTP call
with patch('requests.get') as mock_get:
    mock_get.return_value.json.return_value = {'id': 1, 'name': 'Alice'}
    service = UserService()
    user = service.get_user(1)
    print(user)
    mock_get.assert_called_once_with('https://api.example.com/users/1')

# patch as decorator
@patch('requests.get')
def test_get_user(mock_get):
    mock_get.return_value.json.return_value = {'id': 1, 'name': 'Bob'}
    service = UserService()
    result = service.get_user(1)
    assert result['name'] == 'Bob'

test_get_user()

Interview Mastery

Q:

What is the difference between Mock and MagicMock?

A:

Mock is a basic mock object. MagicMock is a subclass that pre-configures all magic methods (__len__, __iter__, __str__, __enter__, __exit__, etc.) making it work with len(), for loops, with statements, etc. Use MagicMock when the code under test calls magic methods on the mock.

Q:

What is the difference between pytest fixtures and setUp/tearDown?

A:

pytest fixtures: composable (inject multiple), have scope (function/class/module/session), use yield for teardown, can be shared via conftest.py, and are more granular. setUp/tearDown run for every test in the class. pytest fixtures are more flexible and Pythonic.
