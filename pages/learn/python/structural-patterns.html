Structural Patterns

Decorator, Adapter, Observer

Structural patterns compose classes and objects into larger structures.

# Decorator Pattern (structural, not Python decorator)
class Coffee:
    def cost(self): return 2.0
    def description(self): return 'Coffee'

class MilkDecorator:
    def __init__(self, coffee):
        self._coffee = coffee
    def cost(self): return self._coffee.cost() + 0.5
    def description(self): return self._coffee.description() + ' + Milk'

class SugarDecorator:
    def __init__(self, coffee):
        self._coffee = coffee
    def cost(self): return self._coffee.cost() + 0.25
    def description(self): return self._coffee.description() + ' + Sugar'

brew = SugarDecorator(MilkDecorator(Coffee()))
print(brew.description())
print(f'${brew.cost():.2f}')

# Observer Pattern
class EventEmitter:
    def __init__(self):
        self._listeners = {}
    
    def on(self, event, callback):
        self._listeners.setdefault(event, []).append(callback)
    
    def emit(self, event, *args, **kwargs):
        for cb in self._listeners.get(event, []):
            cb(*args, **kwargs)

bus = EventEmitter()
bus.on('user_login', lambda u: print(f'Log: {u} logged in'))
bus.on('user_login', lambda u: print(f'Email: Welcome back {u}!'))
bus.emit('user_login', 'Alice')

# Strategy Pattern
class Sorter:
    def __init__(self, strategy):
        self._strategy = strategy
    
    def sort(self, data):
        return self._strategy(data)

bubble = Sorter(sorted)
quick  = Sorter(lambda d: sorted(d, key=lambda x: -x))

data = [3,1,4,1,5,9,2,6]
print(bubble.sort(data))
print(quick.sort(data))

Interview Mastery

Q:

How do Python's first-class functions affect design patterns?

A:

Many Gang of Four patterns become simpler in Python. Strategy pattern: pass a function instead of a strategy object. Command pattern: use callables. Observer: pass callbacks. Factory: use dict of classes. Python's first-class functions, closures, and decorators replace much boilerplate that OOP patterns require in Java/C++.

Q:

What is EAFP vs LBYL in Python?

A:

EAFP (Easier to Ask Forgiveness than Permission): try/except — Pythonic. `try: x = d['key'] except KeyError: ...`. LBYL (Look Before You Leap): check before acting — `if 'key' in d: x = d['key']`. EAFP is generally preferred in Python since it avoids race conditions and is more readable for common cases.
