Sharding

Sharding horizontally partitions a database across multiple nodes. Each shard holds a subset of data. Enables linear write and read scaling.

📌

Sharding Strategies

1

Hash Sharding

shard = hash(shard_key) % num_shards

Example

customer_id % 4 → shard 0,1,2,3. customer_id=12345 → shard 1

Advantage

Even distribution

Drawback

Range queries span all shards

2

Range Sharding

Shard by value range. Shard 1: A-F, Shard 2: G-M, Shard 3: N-Z

Advantage

Good for range queries

Drawback

Hotspots if data not evenly distributed

⚠️

Drawbacks

  • •Cross-shard joins are expensive or impossible
  • •Re-sharding (adding nodes) requires data migration
  • •Application must know which shard to query

Caching Patterns

Caching stores frequently accessed data in fast memory to reduce compute and database load.

📌

Patterns

1

Cache-Aside (Lazy Loading)

Flow

1. Check cache → 2. If miss: query DB → 3. Store in cache → 4. Return

Drawback

Cache miss = full DB query latency. Cache can have stale data.

General-purpose caching. Cache only what's requested.
2

Write-Through

Flow

Write to cache AND DB simultaneously

Drawback

Write latency = max(cache, DB) latency

Ensure cache and DB always in sync
3

Write-Behind (Write-Back)

Flow

Write to cache → acknowledge → async write to DB

Drawback

Risk of data loss if cache fails before DB write

High-write workloads where DB can lag
💡

Example

ToolRedis
Use CaseCache aggregated daily sales metrics
Codeimport redis r = redis.Redis() # Cache-aside def get_daily_sales(date): cached = r.get(f'sales:{date}') if cached: return json.loads(cached) # Cache miss result = db.query(f'SELECT SUM(amount) FROM orders WHERE date = {date}') r.setex(f'sales:{date}', 3600, json.dumps(result)) # TTL: 1 hour return result

Predicate Pushdown

Push filter conditions (WHERE clauses) to the data source layer — read only the relevant data instead of reading all data and filtering after. Dramatically reduces I/O.

💡

Example

Without PushdownSpark reads ALL 100GB Parquet files from S3 → filters WHERE region = 'East' in memory → 95GB wasted I/O
With PushdownSpark pushes WHERE region = 'East' to Parquet reader → reads only column statistics and row groups where region = 'East' → only reads 5GB
Speedup20x I/O reduction
📌

How Parquet Enables It

  • Row group statistics: min/max per column per row group
  • Column chunk statistics: null counts, distinct counts
  • If WHERE amount > 1000 and row group max_amount = 500, skip entire row group
📌

Z Ordering

Delta Lake ZORDER BY (customer_id) co-locates data with same customer_id → predicate on customer_id skips most files

Vectorized Execution

Process data in batches of vectors (arrays of column values) rather than row by row. Leverages CPU SIMD (Single Instruction Multiple Data) instructions for 10-100x speedup.

📌

Traditional Row By Row

For each row: load customer_id, amount; check amount > 100; add to sum

📌

Vectorized

Load 1024 amounts as array → SIMD: compare all 1024 > 100 in one instruction → sum the vector

📌

Tools Using Vectorization

  • Apache Arrow
  • DuckDB
  • Velox (Meta)
  • Snowflake
  • BigQuery
📌

Apache Arrow

Apache Arrow defines an in-memory columnar format that enables zero-copy data sharing between vectorized engines (Spark, Pandas, Flink, Parquet reader)
