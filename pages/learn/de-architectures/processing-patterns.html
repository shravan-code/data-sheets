Processing Patterns

7 topics

ETL (Extract, Transform, Load)

ETL extracts data from sources, transforms it in an intermediate processing layer (outside the destination), and loads the clean, structured data into the target warehouse. Traditional approach.

📌

Flow

ExtractPull data from sources (RDBMS, APIs, files, Kafka)
TransformClean, validate, aggregate, join in a processing engine (Spark, SSIS, Informatica)
LoadWrite transformed data into destination (warehouse, data mart)
💡

Example

ScenarioDaily sales ETL into Redshift
ExtractAirflow operator runs: SELECT * FROM postgres.orders WHERE date = '{{ ds }}'
Transform{"spark_job":["Cast data types","Join with products and customers","Aggregate to…
LoadWrite final DataFrame to Redshift via COPY from S3

Tools

Apache SparkInformaticaTalendAWS GlueSSISPentaho
⚠️

Drawbacks

  • •Transformation outside warehouse wastes compute — data moved twice
  • •Schema must be defined before loading (rigid)
  • •Long pipeline time — transform must complete before any load
  • •Hard to handle schema evolution in transform layer

ELT (Extract, Load, Transform)

ELT loads raw data into the destination first (data lake or warehouse), then transforms it using the destination's compute power. Modern approach enabled by scalable cloud warehouses.

📌

Flow

ExtractPull raw data from sources as-is
LoadLoad raw data directly into landing zone (S3, Snowflake raw schema, BigQuery staging)
TransformRun SQL/dbt transformations inside the warehouse or on the data lake
💡

Example

ScenarioELT with dbt on Snowflake
Extract LoadFivetran extracts from PostgreSQL → loads raw to Snowflake RAW_DB.ORDERS (no transformation)
Transform{"dbt_models":["stg_orders.sql: SELECT *, CAST(amount AS DECIMAL(10,2)) FROM raw…
OutputSnowflake ANALYTICS_DB.MART.DAILY_SALES ready for BI

Tools

dbtFivetranAirbyteBigQuerySnowflakeDatabricks SQL
📌

Advantages Over Etl

  • Raw data preserved — can re-transform anytime
  • Faster time to first load
  • SQL-based transforms are easier for analysts
  • Leverage massive cloud warehouse compute
⚠️

Drawbacks

  • •Raw sensitive data lands in warehouse — security concern
  • •Warehouse compute costs can be high for complex transforms
  • •Not ideal for very complex transformations (ML feature engineering)

Reverse ETL

Reverse ETL moves processed data FROM the warehouse BACK to operational tools (CRMs, marketing platforms, support tools). Enables data-driven operations by syncing analytics results back to the tools teams use daily.

📌

Flow

Data Warehouse (Gold/Mart layer) → Reverse ETL Tool → Salesforce / HubSpot / Intercom / Zendesk

💡

Example

ScenarioSyncing customer health scores to Salesforce
Warehouse QuerySELECT customer_id, churn_probability, lifetime_value, segment FROM mart.customer_health
DestinationSalesforce CRM — account object updated with churn score field
FrequencyDaily sync
Business ValueSales reps see churn probability directly in Salesforce — can prioritize outreach

Tools

CensusHightouchPolytomicdbt + custom scripts
⚠️

Drawbacks

  • •Risk of overwriting operational data with stale warehouse data
  • •API rate limits of destination tools throttle sync speed
  • •Identity resolution (matching warehouse customer_id to CRM contact_id) is complex
  • •Bi-directional sync creates circular dependency risk

Stream Processing

Processing data continuously as it arrives, event by event or in tiny windows. State can be maintained across events for aggregations, joins, and pattern detection.

📌

Key Concepts

1

Event Time vs Processing Time

Event Time

When the event actually occurred (in the data payload)

Processing Time

When the event was processed by the stream engine

Importance

Out-of-order events cause incorrect results if only processing time is used

2

Stateful Processing

Description

Maintaining state across events (running counts, session tracking, join lookups)

Example

Count user clicks per session — must remember click count as new clicks arrive

3

Windowing

Description

Grouping events into time buckets for aggregation

💡

Example

ScenarioReal-time revenue per product category (5-minute tumbling window)
Flink Pseudocodestream .keyBy(event -> event.category) .window(TumblingEventTimeWindows.of(Time.minutes(5))) .aggregate(new SumAggregator('amount')) .sink(kafkaSink('revenue_by_category'))
Output Every 5min[{"window":"14:00-14:05","category":"Electronics","revenue":45230},{"window":"14…

Tools

Apache FlinkSpark Structured StreamingKafka StreamsAWS Kinesis Analytics
⚠️

Drawbacks

  • •Complex state management and fault tolerance
  • •Late data handling requires watermarks and allowed lateness
  • •Hard to debug and test compared to batch
  • •Expensive always-on infrastructure

Batch Processing

Processing a bounded, finite dataset in a single scheduled run. All data is available upfront. Simpler programming model than streaming.

💡

Example

ScenarioWeekly user cohort analysis
Spark Jobdf = spark.read.parquet('s3://lake/silver/events/year=2024/') cohorts = df.groupBy('cohort_month', 'week_number').agg(countDistinct('user_id').alias('active_users')) cohorts.write.mode('overwrite').parquet('s3://lake/gold/cohort_analysis/')
ScheduleEvery Monday 3am via Airflow
⚠️

Drawbacks

  • •High latency — data only available after batch completes
  • •Resource spikes during batch windows
  • •Failed batches must reprocess entire dataset

Hybrid Processing

Combines batch and streaming in a unified pipeline. Modern lakehouses support both on the same data store. Also called unified batch-stream processing.

💡

Example

ScenarioUnified pipeline using Delta Lake
Streaming PathKafka → Spark Structured Streaming → append to Delta table (Silver) every 30 seconds
Batch Pathdbt runs nightly aggregation on Silver Delta table → Gold table
QueryBI tools query Gold Delta (batch aggregates) AND can also query Silver Delta for recent data

Tools

Apache Flink (DataStream + Table API)Spark Structured StreamingDelta Live Tables
⚠️

Drawbacks

  • •Complexity of managing both processing models
  • •Different latency SLAs for batch vs streaming portions

In-Place Transformation

Transformations happen directly on the data in the storage layer without moving data to a separate processing engine. Common in SQL-first ELT environments.

💡

Example

Scenariodbt transformation in Snowflake
FlowRaw data in Snowflake → dbt runs CREATE TABLE AS SELECT (CTAS) within Snowflake → no data leaves Snowflake
AdvantageZero data movement, leverage Snowflake's MPP compute, SQL-native
⚠️

Drawbacks

  • •Complex ML transforms (feature engineering) hard to express in SQL
  • •Warehouse compute costs for heavy transformations
  • •Limited to SQL — non-SQL logic requires UDFs
