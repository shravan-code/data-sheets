Event Streaming Architecture

A distributed, persistent log of events that acts as the central nervous system of a data platform. Events are immutable facts about what happened. The log is the source of truth.

🔩

Components

1

Event Producers

Applications, microservices, databases (via CDC), IoT devices that emit events

Order service → order_placed eventDebezium → database change events
2

Event Log (Broker)

Persistent, ordered, distributed log of events

Tool

Apache Kafka

3

Stream Processors

Stateful computation over the event stream

4

Sinks

Destinations where processed events land

Delta LakeElasticsearchCassandraPostgreSQLAnother Kafka topic

Message Queue Architecture

A message queue enables asynchronous communication between producers and consumers. Each message is consumed by exactly ONE consumer. Designed for task distribution and work queues.

📌

Difference From Streaming

Message QueuePoint-to-point. Message consumed once and deleted. No replay. (RabbitMQ, SQS)
Event StreamingPublish-subscribe. Event persisted. Multiple consumers. Replayable. (Kafka, Kinesis)
💡

Example

ToolAWS SQS
Use CaseImage processing queue
Flow["User uploads image → S3 event → SQS message added","Worker pool pulls messages…
⚠️

Drawbacks

  • •No replay — once consumed, message is gone
  • •Single consumer per message — not suitable for fan-out
  • •No built-in time-series analytics

Pub/Sub Architecture

Publishers send messages to topics without knowledge of subscribers. Multiple subscribers independently receive all messages from the topic. Decouples producers from consumers.

💡

Example

ToolGoogle Cloud Pub/Sub
Use CaseMulti-consumer event fan-out
Topicorder-events
Subscriptions["analytics-sub → BigQuery streaming insert","inventory-sub → inventory service"…
NoteEach subscription maintains its own offset — analytics can be behind without affecting notification service
⚠️

Drawbacks

  • •No ordering guarantee across messages (in most pub/sub systems)
  • •At-least-once delivery requires idempotent consumers
  • •Complex to debug fan-out when one subscriber is slow

Stream-Table Duality

In Kafka, a stream and a table are two views of the same data. A stream is an infinite sequence of events. A table is a snapshot of the current state (materialized view of the stream). Kafka Streams and ksqlDB implement this duality.

📌

Duality

Stream To TableAggregate a stream of events → current state table (running sum, latest value per key)
Table To StreamCapture changes to a table → stream of change events (CDC)
💡

Example

Streamorder_events topic: [order_placed, order_shipped, order_returned]
Table Derivedcurrent_order_status table: {order_id: 123, status: 'shipped'}
KsqlCREATE TABLE order_status AS SELECT order_id, LATEST_BY_OFFSET(status) AS status FROM order_events GROUP BY order_id EMIT CHANGES;

Stateful vs Stateless Processing

Stateless processors transform each event independently. Stateful processors maintain state across events (counts, sums, session data, joins).

📊

Comparison Table

Featurestatelessstateful
Memory neededNone (event-local)State store (RocksDB in Flink)
ScalabilityTrivially parallelizableMust route same key to same partition
Fault toleranceSimple restartState must be checkpointed
ExamplesFilter, map, format changeAggregation, join, session detection
📌

Stateful Example

Use CaseCount orders per customer in last 1 hour
Flinkstream .keyBy(e -> e.customerId) .window(SlidingEventTimeWindows.of(Time.hours(1), Time.minutes(5))) .aggregate(new CountAggregator())
State StoreFlink stores per-customer count in RocksDB. Checkpointed to S3 every 60 seconds for fault tolerance.

Exactly-Once Semantics

Guarantees that each event is processed exactly once — no duplicates, no data loss. Hardest guarantee to achieve in distributed systems. Requires coordination between producer, broker, and consumer.

📌

Delivery Semantics

1

At-Most-Once

Fire and forget. May lose events. Never duplicates.

Metrics where occasional loss is acceptable
2

At-Least-Once

Events may be delivered multiple times. No data loss.

Most use cases with idempotent consumers
3

Exactly-Once

Each event processed exactly once. No loss, no duplicates.

Financial transactions, billing, inventory
📌

Kafka Exactly Once

Producer SideEnable idempotent producer: enable.idempotence=true. Each message has unique sequence number — broker deduplicates.
Consumer SideKafka Streams / Flink checkpointing: offset committed atomically with output write using Kafka transactions.
Flink Configenv.enableCheckpointing(60000); // checkpoint every 60s env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
⚠️

Drawbacks

  • •Performance overhead (coordination, checkpointing)
  • •Requires all components to support transactions
  • •Not needed for all use cases — at-least-once with idempotent sinks is often sufficient

Windowing Strategies (Tumbling, Sliding, Session)

Windows group events into finite sets for aggregation in streaming pipelines.

📌

Window Types

1

Tumbling Window

Fixed-size, non-overlapping windows. Each event belongs to exactly one window.

Flink

TumblingEventTimeWindows.of(Time.minutes(5))

2

Sliding Window

Fixed-size, overlapping windows. Window slides by a specified interval.

Flink

SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5))

3

Session Window

Dynamic windows based on activity gaps. Window closes after inactivity gap.

Flink

EventTimeSessionWindows.withGap(Time.minutes(30))

4

Global Window

Single window containing all events. Used with custom triggers.

Use Case

Trigger processing after N events, not on time

Watermarking & Late Data Handling

Events often arrive out of order (network delays, mobile offline sync). Watermarks tell the stream processor how far behind real time it should expect events, allowing windows to close correctly.

📌

Watermark

DefinitionA timestamp that says: 'I guarantee no event with timestamp < watermark will arrive'
ExampleWatermark = max_event_time - 2_minutes. If latest event is at 14:10, watermark is at 14:08. Windows with end_time <= 14:08 can safely close.
Flink Configstream.assignTimestampsAndWatermarks( WatermarkStrategy .<Event>forBoundedOutOfOrderness(Duration.ofMinutes(2)) .withTimestampAssigner((e, ts) -> e.getTimestamp()) );
📌

Late Data Strategies

1

Discard late events

Events arriving after watermark are dropped

When late events are rare and correctness is less critical
2

Allowed Lateness

Window stays open for an extra duration after watermark passes, accepting late events and updating results

Flink

.allowedLateness(Time.minutes(5))

3

Side Output for Late Events

Route late events to a separate stream for separate handling (reprocessing queue, alerting)

Flink

.sideOutputLateData(lateOutputTag)
